# Kubexm 全链路产品级测试方案 (v3.0 终极详尽版)

## 一、 测试总体架构与核心定位

### 1.1 系统定位分析
`kubexm` 作为一款企业级 Kubernetes 集群全生命周期管理工具，核心卖点在于 **完全离线优先 (Air-Gapped First)**、**双部署模式 (Kubeadm & Kubexm Binary)**、以及 **高度解耦的模块化设计**。系统测试必须确保从最上层的 CLI 命令分发到最底层的 SSH 执行流，完全契合防御性编程理念，即使遭遇极端网络、IO 或人为配置失误，仍能保证不破坏集群一致性。

### 1.2 层级职责校验准则 (Layered Architecture Validation)
针对 `REFACTOR_GUIDE.md` 中的层级隔离要求，方案引入以下强制性审计标准：
1. **Pipeline 规划层审查**: Pipeline 唯一指责为组装 Module 并提供全局异常捕捉(`SafePlan`)。所有未声明 `--dry-run` 的流程在真正触发 `Engine.Execute` 之前，不可以产生任何对目标机器的 IO 写入。
2. **Module & Task 隔离**: Module 必须负责生成 `ExecutionFragment` 并正确解决图冲突 (`MergeFragment`)。Task 只能进行同组件相关的原子 `Step` 排列（如 `InstallKubelet` 只能编排 `BinaryDownload`、`SystemdConfig` 等 Step），严禁跨域编排。
3. **Step 原子性与隔离性**: 全局 600+ Step 文件**严禁**直接 Import 或调用 `internal/connector` 逻辑。必须依托于 Runner 层透传的接口调用目标节点。
4. **Runner 引擎**: 负责拦截全部超时 (`Timeout`) 并在发生致命错误时驱动已执行 Step 触发 `.Undo()` (如果支持)。

## 二、 运行模式隔离与自适应测试策略

### 2.1 堡垒机大网通信原则 (Bastion Host Principle)
测试强制向所有的输入源 (`config.yaml`, `host.yaml`) 混入环回地址 (`127.0.0.1`, `localhost`)，检验 Preflight 阶段的安全阀门能否成功将其拦截，确保所有的 SSH 管理通道必须基于物理大网 IP 与受控凭据。

### 2.2 离线自适应感知网络 (Offline Automatic Sensing)
- **在线模型 (Online)**: 测试机具有互联网出口，执行 `create cluster` 能够顺利通过内置工具拉取 GitHub / 镜像源的必要数据。
- **孤岛模型 (Air-Gapped)**:
  - 切断目标主机与外网的所有连接。
  - 在有网主机利用 `kubexm download --config cluster.yaml -o packages/` 将产物拖拽至特定目录。
  - 将 `packages/` 移植到无网主机，引擎探测到该路径时必须开启“离线分发模式”，将内部打包的 Tar 归档及容器镜像直接通过 SSH/SFTP 推送到各节点，无需依赖包管理器。

## 三、 测试方法学与阶段深度剖析

### 3.1 静态检查与单元测试 (Static & Unit Testing)
- 工具链：`golangci-lint`。强制拦截环状依赖 (Cycle Import)。
- 对 `config.yaml` 的 YAML 解析进行全边界值的 Table-Driven Test，确保如 `loadbalancer.mode` 等非法输入在反序列化期间就能默认容错处理。
- `ConnectionPool` 并发测试：利用 `go test -race` 验证对大规模主机发起高并发连接时的并发安全性。

### 3.2 DAG (有向无环图) 正确性断言
- 利用 `--dry-run` 标志对全部 23 个系统部署矩阵进行仿真跑流。
- 断言图路径：提取生成的输出流，确认节点依赖关系无死锁。例如：`ControlPlaneModule` 中的 `StartAPIServer` Step 必须拥有向前的 `InitETCD` 依赖。

### 3.3 E2E (端到端集成) 全矩阵演练
1. **多 OS 兼容**: 从 Ubuntu 20/22 到 RHEL 体系的 CentOS 7/8, Rocky 8/9, AlmaLinux 进行笛卡尔积交叉测试。
2. **全流程巡检**: 采用真实的 IaC 工具 (如 Terraform) 拉起目标实例组，顺次执行 `create` -> `scale` -> `reconfigure` -> `certs renew` -> `delete`。
3. **断言手段**: E2E 终态验证不再依靠 `kubexm` 的控制台输出，而是通过标准的 Kubernetes 一致性验证组件 (Sonobuoy) 或 `kubectl get componentstatuses` 来进行第三方确认。

### 3.4 容灾、幂等与故障注入 (Idempotency & Chaos)
- **绝对幂等**: 对于一个状态为 `Running` 的集群，重复触发 `cluster create` 命令。预期结果为：控制流急速通过，底层检测全部变更为 `[Skip]`，集群控制面无重启时间（Uptime 不变）。
- **Chaos 断流测试**: 模拟执行流到 40% (如 `ControlPlane` 安装期间) 强行断电或 `kill -9` 关闭主进程。后续利用 Checkpoint 文件重新唤醒，检验从断点恢复的可靠性。
- **部分组件破坏恢复**: 故意清理特定节点的证书或删除特定 static pod manifest，利用 `restore` 功能测试其补偿机制。
