# Kubexm 全景式质量保障与架构验证方案 (v6.0 终极白皮书版)

## 一、 系统定位与测试核心哲学

### 1.1 `kubexm` 产品定位
`kubexm` 是针对企业级完全隔离环境（Air-Gapped First）设计的 Kubernetes 离线生命周期编排器。其不局限于简单的 `kubeadm` 封装，更原生支持了基于纯二进制（Systemd）和原生 Kubeadm 双重并行栈，外加包含 Keepalived、Haproxy、Nginx 及 Kube-Vip 在内的复杂拓扑外围系统。这是一套涉及 OS 底层（Iptables, CRI, 证书体系）的高危自动化资产，故需要顶格（Top-Tier）的测试防御策略。

### 1.2 绝对防御原则 (The Golden Rules)
1. **指令与交互 100% 覆盖**：包含所有的 Cobra 指令 (`Use`)，缩写参数，隐藏参数，交互打断机制（模拟点击 `y/N`）。
2. **幂等性绝对防御 (Idempotency)**：
   测试要求系统呈现“状态收敛”。任意操作如 `cluster create` 或 `cluster upgrade`，只要目标机器没有发生退化，反复执行必须获得 `[Skip]` 标签，且核心业务 Uptime 不能受到重启影响。
3. **架构纪律审计 (Architectural Discipline)**：
   **强制规定**：底层逻辑包 `internal/step/` 严禁产生任何 `import "github.com/mensylisir/kubexm/internal/connector"`，防止产生脏连接导致并发泄漏。所有的底层命令下发必须借助 `Runner`。
4. **控制流断点快照可恢复性 (Checkpoint Recovery)**：
   模拟 `kill -9` 强行关闭执行主控机，测试引擎需利用落盘缓存无缝衔接回断点继续安装。

## 二、 架构隔离与自适应感知测试策略

### 2.1 堡垒机原则的物理级防越权 (Bastion Host Principle)
向配置中心 (`host.yaml`, `config.yaml`) 混入环回口 (`127.0.0.1` / `localhost`) 等非法节点池。系统的 `PreflightModule` 必须第一时间进行安全熔断，强制要求通过真实的大网 IP 与完整的 SSH 密钥链通讯。测试这一隔离墙能否 100% 抵御恶意部署输入。

### 2.2 离线自感知与环境自适应 (Offline Sensing Context)
- **在线模式 (Online Flow)**: 测试在普通公网环境下，自动解析 `amd64`/`arm64` 并向 Github 和镜像池发起高并发直连下载并完成部署。
- **孤岛模式 (Air-Gapped Flow)**:
  核心验证路径。完全切断内外网后，仅提供由 `download` 命令吐出的 `packages/` 离线弹药库。系统应**不报任何一层 Network Error**，立刻触发 Offline Sensing 机制，改为内网高速 SFTP 分发与 Local 镜像 Push，实现无痛脱网部署。

## 三、 测试方法学与阶段深度剖析

### 3.1 静态探针与白盒审计 (Static & Unit)
- 工具链：`golangci-lint`。强制拦截任何未被包装的原始 `panic`，拦截所有的模块环状引用 (Cyclic Import)。
- 对 `config.yaml` 中涉及到的海量配置项实行 Fuzzing 级别容错，非法字段（如不被支持的 `loadbalancer.mode`）需在反序列化期间提供 Fallback 或快速抛错。
- **并发调度池 (`ConnectionPool`)**: 利用 Go 的 `-race` 条件测试模拟针对上百节点的并发 `SSHConnector` 连接拿取，避免死锁或协程悬挂。

### 3.2 逻辑沙盘与图断言 (DAG Dry-Run Probing)
所有的写操作（`create`, `delete`, `upgrade`, `reconfigure`）必须包含 `--dry-run` 用例。
在沙盘模式下，`Engine` 只计算、渲染 `ExecutionFragment` 并输出合并后的单向无环图，断言每个节点（如 `NetworkModule` 依赖 `ControlPlaneModule`）时序准确且绝不溢出任何机器写操作。

### 3.3 混沌与灾难注入演练 (Chaos Engineering)
1. **脑裂复原 (Etcd Split-Brain Recovery)**: 人为在 `M01` 拓扑中搞坏证书或关闭 ETCD 容器，使得 API Server 完全下线。随后调用 `cluster restore` 读取快照，验证从“尸体”直接原位满血复活的能力。
2. **凭据安全热更新 (Zero-Downtime Cert Rotation)**: 对即将到期的叶子证书执行 `certs renew kubernetes-certs`，验证热刷证书并不重启根信链，集群业务连线平滑过渡。
3. **系统崩溃捕获 (SafePlan Guard)**: 故意改烂某内部组件使得模块 Plan 阶段遭遇空指针，要求系统不能暴露丑陋冗长的 Panic Stacktrace，而是由 `pipeline.SafePlan` 温和接管并上抛 `internal error`。

## 四、 测试用例层级标准与出口管控
- **Code Coverage**: `internal/util/`, `config/`, `pipeline/` 的纯逻辑单元覆盖率要求 >85%。
- **E2E 成功率**: 集群全生命周期（M01 到 M23）部署、添加/卸载 Worker、更新证书并完全清理退出的 E2E 测试成功率要达到 100%。所有资源的正确性不凭日志，一律凭 `kubectl get componentstatuses` 断言。
