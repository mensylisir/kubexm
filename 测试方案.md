# Kubexm 核心架构与多维生命周期全量测试方案 (v5.0 终极白皮书)

## 一、 系统概览与全景测试模型

### 1.1 系统核心业务定位
`kubexm` 是一款基于 Go 语言开发的企业级 Kubernetes 基础设施编排器。它摒弃了传统的脚本式部署，采用**声明式状态机 (Declarative State Machine)**、**有向无环图编排 (DAG Orchestration)** 和**原子化操作 (Atomic Operations)**。
核心亮点在于：
1. **Air-gapped First (原生离线模式)**：完全不依赖外部网络、OS 包管理器，所有二进制和镜像全部依靠自带分发通道。
2. **Kubeadm/Kubexm 二元融合部署**：支持原生 Kubeadm 标准部署，或完全剥离 Kubeadm 的纯系统级（Systemd）二进制安装。

### 1.2 全景分层测试金字塔
根据 `REFACTOR_GUIDE.md` 强制划分的边界，测试方案必须从上至下实施“严防越权”审计。
- **CLI 层 (Cobra Commands & Flags)**: 负责 100% 覆盖所有的交互拦截（Confirm Buttons）、全局标志(`--yes`, `--verbose`, `--dry-run`)、参数重载与缺失校验。
- **Pipeline 调度编排层**: 针对 `internal/pipeline/` 下的 22 种 Pipeline（如 `CreateClusterPipeline`, `RenewEtcdCAPipeline`），核心测试其装配 Module 序列的正确性，以及 `SafePlan` 包装器能否正确兜底 Panic。
- **Module & Task 逻辑域**: 测试针对不同 `kubernetes.type`, `etcd.type` 和 `loadbalancer` 所产生的分支 (ExecutionFragment) 组合，能否正确解析 `host.yaml` 并转换为合并后的有向无环图节点。
- **Step 原子动作层**: 必须保证这 600+ 个 Step 的代码内部，不存在任何绕开 Runner 层直接向目标机器发起 SSH/SFTP 或 Shell 执行的违例代码。并且所有的 Step 都具备先决条件探测 (`check()`) 以支持幂等。
- **Runner / Connector 执行网关层**: 测试并发连接池 (`ConnectionPool`) 的死锁预防、对大网物理 IP 强制绑定（拦截 localhost）、超时、重试策略 (`Retries`) 和基于 Context 的优雅退出。

## 二、 核心测试策略论 (Testing Methodologies)

### 2.1 静态与架构纪律审查 (Architecture & Code Review)
1. **反向依赖扫描**: 在 CI 中嵌入 `grep` / `golangci-lint`，强校验 `internal/step/` 包**绝对禁止** `import "github.com/mensylisir/kubexm/internal/connector"`。
2. **环回地址隔离法 (Bastion Principle)**: 分析 `Preflight Connectivity` 模块的拦截逻辑，对于传入的 `127.0.0.1` 或是本地主机名，必须立刻返回严重错误。

### 2.2 API 与配置解偶测试 (API & Config Matrix)
1. **反序列化边界容错**: 准备一组具有极端残缺字段的 `config.yaml`（例如缺少 `arch`），验证其反序列化引擎是否具备默认降级（回退为 `amd64`）和默认值填充。
2. **K8s Client-Go API 交互仿真**: 针对 `node` 相关的子命令，提供一套伪造的 `kubeconfig` 或对真实 API Server 发起读写操作，捕捉并验证对 `kubectl` 行为的包装正确性。

### 2.3 混沌与容灾演习 (Chaos & Fault Tolerance)
1. **网络抽离 (Network Partition)**: 测试引擎的离线智能感知能力。向无外网工作区挂载 `download` 产出的 `packages/`，工具应全程不产生网络报错并安装成功。
2. **断点快照 (Checkpoint Recovery)**: 模拟在 `WorkerModule` 执行中段实施操作系统层面的进程 `kill -9`，重新唤醒时验证工具基于快照精确恢复状态图，从断点直接继续的能力。

### 2.4 性能及伸缩极限 (Performance & Scalability)
- **并发连接池压测**: 配置超过 150 个节点的假主机清单，跑查 `kubexm` 对 `Connector Pool` 的调度管理，避免触发系统层的 `Too many open files (ulimit)` 报错。
- **执行时间墙 (Timeout)**: 所有的任务流从始至终均有 30~60 分钟超时的紧箍咒（在 Pipeline 基础层定义）。需人为触发长耗时任务（如卡住的 systemctl restart）以校验超时传递。
