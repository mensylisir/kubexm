# Kubexm 全维产品级测试方案 (v4.0 终极详尽覆盖版)

## 一、 系统定位与测试哲学

### 1.1 工具定调与愿景
`kubexm` 是一款企业级的 Kubernetes 离线生命周期管理 CLI 平台。它主打**纯离线/孤岛环境支持 (Air-Gapped Support)**、**全双工安装模式 (Kubeadm 静态 & 纯二进制 Kubexm)**，并内置了极其丰富的负载均衡和 ETCD 同步拓扑方案。由于它是一款在操作系统底层直接操纵证书、服务和网络的 IaC 式引擎，质量红线要求极高。

### 1.2 测试基调与绝对准则 (The Golden Rules)
1. **幂等性绝对防御 (Idempotency)**：
   测试核心理念为“防抖与容灾”。无论是节点级 API 还是集群级 Pipeline 操作，在没有任何目标配置变动的情况下，重复执行必须是 `No-Op` (跳过执行并瞬间返回)，且绝对不允许引入状态污染（Dirty State）。
2. **越权阻断与防爆规避 (Blast Radius Containment)**：
   - 根据代码库中 `ARCHITECTURE_PLAN.md` 约束，严禁系统在执行阶段对不在 `host.yaml` 名单中的设备（尤其是通过 `127.0.0.1`）执行破坏性 SSH。测试必须引入强制校验。
3. **架构纪律审计 (Architectural Purity)**：
   - **Pipeline** 仅用于组装 `Module`。
   - **Module** 仅产生并化解 `ExecutionFragment` 的图依赖。
   - **Task** 由纯正的 `Step` 拼接。
   - **Step** 层**严禁**任何绕过 `Runner` 直接调用 `Connector` 的越级流氓调用。测试方案将强制推行这一代码结构的自动化扫描。

## 二、 全量被测对象拆解 (Test Objects Breakdown)

### 2.1 交互界面层 (CLI Interface & Buttons/Prompts)
- **命令树 (Cobra Command Tree)**: 包括 `cluster`、`node`、`certs`、`registry`、`download`、`iso`、`images`、`config` 以及所有的下挂子命令。
- **动态终端拦截与交互 (Terminal Prompts)**: 所有在缺少 `--yes` (或 `-y`) 时的确认悬停（即相当于前端的“确认按钮”和“取消按钮”输入流拦截）。
- **配置数据源注入**: `host.yaml` (主机清单与拓扑凭据)、`config.yaml` (Kubernetes 特性声明，例如 `loadbalancer.mode` 等 23 种子图条件)。

### 2.2 逻辑调度与图执行层 (DAG Orchestration & API)
- **模拟渲染 (`--dry-run`)**: 不下发真实 I/O 动作的图逻辑编排测试，验证 DAG（有向无环图）无死锁、无回环。
- **Kubernetes 交互 API**: 在执行完环境就绪后，通过 `internal/task/kubernetes/` 包内的模块对 Kubernetes API Server 施加实际探测 (如 `node list`, `cluster health`)。

### 2.3 底层资源分发 (Connection Pool & File Systems)
- **网络模式自适应**:
  - **离线网络探知**: 核心测试路径。测试在纯封闭内网下，将 `packages/` 资源喂给引擎后，其内部镜像推入和二进制分发逻辑是否自动激活。
  - **SSH/Local 并发池**: 对超大规模节点 (100+ Nodes) 并发执行命令时，连接池调度器能否避免超时（OOM / Too Many Open Files）。

## 三、 测试方法论与实施工具

### 3.1 静态检查与深度白盒分析 (Static Code Analysis)
利用 `golangci-lint` 及人工 Code Review 拦截如下违规操作：
- `import` 引用链倒置检测：一旦发现 `internal/step/*` 下有引入 `github.com/mensylisir/kubexm/internal/connector`，测试流立即抛错阻断，维护代码解耦纯洁性。
- YAML 映射安全兜底测试：强制校验配置文件缺少必要字段时（如空 `arch` 必须被反射为 `amd64`，非法 `loadbalancer.type` 必须在 Plan 阶段拦截）。

### 3.2 并行端到端演练 (Concurrent E2E & Matrix Execution)
- 依靠 VMWare ESXi / KVM 构建不同 OS 的异构靶场环境。
- **23 部署矩阵 (The 23-Branch Matrix)**: 对应产品需求中的 `M01-M23`，针对这 23 套完整的负载均衡、ETCD和K8s组合，必须每一套均经过“初始化 -> 断电恢复 -> 节点扩容 -> 重新生成证书 -> 备份 -> 清除集群”全套 E2E 考验。
- 断言：不再依赖工具本身的输出成功，必须用 `kubectl get componentstatuses` / `kubectl get nodes` 或第三方 `Sonobuoy` 作为一致性法官。

### 3.3 容灾注入与边界探错 (Chaos Engineering)
1. **网络闪断与大风暴**: 在拉取 `packages` 或 `images push` 的途中掐断网络接口，期望工具不挂起死锁，而是依靠 `Runner` 输出 `Timeout` 并自动进入回退 (`Undo`)。
2. **Panic() 防护伞 (`SafePlan`)**: 定制含有 `panic("test")` 探针的二进制文件放入 `Module`。确认工具绝不允许输出底层的 Go 堆栈导致控制台脏乱，而是由全局的 `SafePlan` 包装器转化为柔性的友善报错日志安全退出。
