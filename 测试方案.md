# Kubexm 全链路产品级测试方案 (v2.0 终极版)

## 一、 测试目标与系统定位

### 1.1 系统定位
`kubexm` 是一款企业级、离线优先的 Kubernetes 集群生命周期管理 CLI 工具。它支持灵活的 Kubernetes 安装类型（`kubeadm` 标准模式与纯 `kubexm` 二进制模式）、混合 Etcd 模式以及丰富的负载均衡拓扑。

### 1.2 核心测试目标
本方案致力于建立一个无死角、自动化的保障体系，目标是：
1. **指令 100% 覆盖**：覆盖 `cmd` 下定义的所有全局参数、子命令和特定 flag。
2. **状态幂等 100% 保障**：对同一个集群拓扑重复执行任意命令，不应导致状态漂移（State Drift）或服务雪崩。
3. **架构规范 100% 遵从**：自动化审计代码，确保不存在越级调用（例如 Step 绕过 Runner 直接调用 Connector 的 SSH 方法）。
4. **异常容灾覆盖**：模拟真实的物理断网、进程 Crash 和 SSH 连接闪断场景。

## 二、 架构规范与边界测试策略

根据 `ARCHITECTURE_PLAN.md` 和 `REFACTOR_GUIDE.md` 的强制要求，测试需重点验证以下边界：

### 2.1 架构分层职责校验 (Layered Architecture Validation)
测试过程中通过代码静态扫描与执行树分析验证：
- **Pipeline 层**: 只能负责组装 `Module` 并捕捉全局 Panic (`SafePlan`)。测试需模拟传入非法的 `Module` 验证拦截。
- **Module 层**: 只能负责跨主机的逻辑域封装并返回 `ExecutionFragment`。
- **Task 层**: 必须是由一个个 `Step` 组装的有向无环图，不做任何具体机器操作。
- **Step 层**: 强制原子化。测试需针对 600+ Step 脚本注入回退逻辑（Undo 测试），验证状态补偿。
- **Runner 层**: 控制中心。在此注入连接延迟与失败，测试其内建的 Retry、Timeout 和 Context 传递机制。

### 2.2 中心机器原则测试 (Bastion Host Principle)
- 强制拦截：系统必须检测 `host.yaml` 中是否存在 `127.0.0.1`，即便工具与目标集群节点共驻一机，也必须强制通过大网 IP 和标准 SSH 鉴权，测试将故意混入环回地址以验证该拦截阀。

### 2.3 离线与在线模式自适应测试
- **离线感知**: 剥夺执行节点的外部路由，在工作区放置通过 `download` 产出的 `packages` 压缩包。执行引擎必须能够探测并切换为纯本地化镜像推入和 RPM/DEB 投递。
- **防污染校验**: 下载或解压的文件必须局限在当前 Cluster 的工作目录（例如 `${workdir}/${cluster_name}/`）下，杜绝跨集群覆盖。

## 三、 测试方法学与阶段设计

### 3.1 核心单元测试 (Unit & Static)
使用 Go `testing` 包，重点覆盖 `config` 解析（容错及默认补全）、`util/` 工具函数。涉及 I/O 的方法使用 `t.TempDir()` 或 `afero` 内存文件系统。静态扫描工具严查循环依赖。

### 3.2 有向无环图断言 (DAG Dry-Run Testing)
所有改变系统状态的操作（create, delete, upgrade, reconfigure）必须提供完整的 `--dry-run` 用例。
在 CI 管道中捕获 `--dry-run` 输出的图结构，断言其依赖链路（例如 `LoadBalancerModule` 必须在 `ControlPlaneModule` 之前规划完毕）是否偏离基准快照。

### 3.3 端到端环境矩阵验证 (E2E Matrix Automation)
借助 Terraform 或 Vagrant 动态拉起靶机。靶机群涵盖单架构与混合架构，执行生命周期指令，最后利用 Sonobuoy 或直接 `kubectl get nodes` / `pods -n kube-system` 断言底层资源的一致性。

## 四、 性能与极限场景测试

1. **连接池风暴测试**: 对 `Connector` 中的 `ConnectionPool` 施加并发连接，模拟 200 个节点的集群分发场景，验证 `MaxPerKey` 机制，确保不会遇到 `too many open files` 异常。
2. **长效执行中断测试 (Checkpoint Recovery)**: 集群部署至 `WorkerModule` 时强制杀死进程（`kill -9`），验证下次启动时 Checkpoint 机制的图复原与恢复精度。

## 五、 测试度量标准
1. **零崩溃准则**: 不论配置文件如何残缺，CLI 工具绝不允许出现未被捕获的 Panic。
2. **100% 环境销毁**: 集群 Delete 后，系统内不应当存留 iptables 污染、僵尸容器或未卸载的虚拟网卡。
