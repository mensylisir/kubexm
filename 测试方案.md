# Kubexm 产品级测试方案

## 一、 测试目标与原则

### 1.1 测试目标
本方案旨在为 `kubexm` 集群生命周期管理工具提供极其详尽、无死角的质量保障策略。工具的定位是支撑企业级离线优先部署、双模式创新（Kubeadm + Kubexm 二进制）的集群安装配置工具。测试的终极目标是保证系统在各类复杂网络环境（尤其是完全离线）、复杂底层架构和操作系统中，都能具备极高的稳定性、容错性和状态幂等性。

### 1.2 核心测试原则
1. **防破坏与幂等原则**：所有执行动作（Step 层）必须是原子的、幂等的。多次对同集群执行相同操作必须得到一致结果，且不能产生垃圾数据或引发组件崩溃。
2. **架构边界校验原则**：严格监督层级职责。严禁跨层调用（例如 Step 直接调用 Connector），测试过程将通过代码审计和自动化执行流监控来保障这一约束。
3. **安全与合规优先**：禁止使用环回地址（localhost/127.0.0.1）作为管理节点。所有集群内部的交互必须通过宿主机大网 IP 及合法的 TLS/SSH 凭证。
4. **离线优先保障**：网络隔离下的集群部署和升级是产品的核心竞争力，方案需模拟完全气隙(Air-Gapped)网络环境下的各项行为。

## 二、 被测对象架构及特征分析

### 2.1 系统架构分层解析
- **CLI 层 (cmd/)**: 接收终端输入，使用 Cobra 构建。关注点在于参数校验、默认值补全。
- **Pipeline 层**: 最顶层的业务流编排（如 CreateCluster, Backup, Restore 等）。关注点在于模块加载顺序、以及 `SafePlan` 异常捕获机制。
- **Module 层**: 功能域模块（如 ETCD Module, ControlPlane Module）。
- **Task 层**: 具备业务含义的动作组（如 UpgradeKubelet）。关注点是组装 Step。
- **Step 层**: 最小执行单元。关注点是幂等性检测和单一职责。
- **Runner 层**: 控制 Step 生命周期的引擎（Plan -> Execute -> Undo），包含重试、容错机制。
- **Connector 层**: SSH 及本地连接底层封装。关注连通性与并发连接池 (`ConnectionPool`) 的稳定性。

### 2.2 核心特性关注点
- **集群架构多样性**：支持单主/多主。
- **部署类型交叉支持**：
  - K8s: `kubeadm` vs `kubexm` (二进制)
  - Etcd: `kubeadm` vs `kubexm` vs `exists` (外部接入)
- **负载均衡拓扑**：支持 internal(静态pod/二进制)、external(keepalived+haproxy/nginx) 以及 kube-vip。

## 三、 测试策略与方法学

### 3.1 静态与代码审查 (Static & Code Review)
- **依赖与循环调用检查**：验证项目（特别是 utils 目录下）不存在严重的循环依赖（Cycle Import）。
- **层级依赖审计**：确保 `internal/step/` 目录下的 600+ 文件没有任何一处直接引入或使用 `internal/connector` 的方法。
- **配置隔离审查**：确认所有的 YAML 模板解析逻辑在 `templates/`，而业务逻辑在 `lib/`，保证无状态特性。

### 3.2 单元测试策略 (Unit Testing)
- **核心逻辑单元测试**：针对 `config` 配置解析器、`runner` 的状态机流转进行 100% 覆盖。
- **沙盒文件系统模拟**：所有操作 OS 文件、路径生成的测试（如 `local_connector`）必须使用 `t.TempDir()` 或 Mock 接口，防止污染测试运行机。
- **故障注入 (Fault Injection)**：在 Connector 层 Mock 网络超时和认证失败，验证 Runner 的超时时间（Timeout）、重试次数以及取消信号（Context Propagation）是否正确生效。

### 3.3 接口与有向无环图测试 (DAG / Integration)
- **模拟运行测试 (`--dry-run`)**：
  不建立真实的 SSH 连接，仅根据传入的 `config.yaml` 生成执行图 (ExecutionGraph)。断言 DAG 中各个 Node (Task/Step) 的先后依赖关系是否正确。
  例如：*网络插件部署 Node 必须依赖于 Kube-APIServer 就绪 Node*。
- **安全拦截校验**：验证由于不支持的操作导致的报错拦截，例如 `UpgradeEtcdPipeline` 应在 Plan 阶段抛出 `NotImplemented` 错误，禁止其生成错误的图。

### 3.4 端到端集成与验收测试 (E2E)
- **环境搭建**：构建多 OS 混合集群（例如 Master 为 Rocky Linux，Worker 为 Ubuntu）。
- **主流程验证**：基于测试矩阵（23个分支），执行全链路操作。验证过程不仅要看命令返回值，必须使用 `kubectl` 等原生客户端深入被测机器内，获取 Pods、DaemonSet 和 Certificates 的实际状态。

### 3.5 非功能性测试
- **容灾与中断测试**：在安装/升级进行到 50% 时，手动中止 `kubexm` 进程，重新执行，验证集群是否能从断点恢复并正确完成（依赖 Checkpoint 机制）。
- **性能与压力测试**：并行对 100+ 个节点的规模进行 Worker 添加。验证 Connector 的 SSH 连接池 (`ConnectionPool.MaxPerKey`) 是否会引发句柄耗尽或死锁。

## 四、 测试环境与资源规范

### 4.1 测试机群模型 (Testbeds)
根据不同测试需求配置三种模型环境：
1. **MinT (Minimum Topology)**：1 Master, 1 Worker（快速回归，单元测试）。
2. **HAT (High Availability Topology)**：3 Master, 2 Worker, 2 LB Nodes（完整架构验证，E2E 测试）。
3. **MasT (Massive Topology)**：3 Master, 100+ Mocked Workers（并发和压力测试）。

### 4.2 操作系统覆盖范围
- **RHEL 家族**: CentOS 7.9, Rocky Linux 8/9, AlmaLinux 8/9
- **Debian 家族**: Ubuntu 20.04 LTS, Ubuntu 22.04 LTS

### 4.3 网络拓扑要求
- **Online Mode (联网区)**：允许通往外网，用于执行 `kubexm download`。
- **Air-Gapped Mode (孤岛区)**：纯内网，禁止任何形式的外网路由，DNS 解析指向内部 CoreDNS。所有集群安装测试必须在该区域完成。

## 五、 测试度量与通过标准
- **代码覆盖率**：核心模块 (`internal/pipeline`, `internal/engine`, `internal/runner`) 单元测试行覆盖率 >= 85%。
- **E2E 成功率**：核心执行分支（测试矩阵 M01-M21）执行成功率 100%。
- **重试恢复率**：在模拟中断容灾测试中，二次执行恢复成功率 >= 98%。
- **零安全违规**：禁止通过 `127.0.0.1` 操控本机节点，全部遵循堡垒机通过大网口分发的纪律。
