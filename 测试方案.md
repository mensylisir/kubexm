# Kubexm 测试方案

## 1. 测试目标
本测试方案旨在为 `kubexm` 命令行工具提供全面的质量保障策略。确保在不同的部署架构、离线/在线环境、以及所有支持的命令分支下，集群的创建、管理、销毁和升级等操作具备高度的稳定性、幂等性和原子性。

## 2. 测试范围概述
- **CLI 命令层**: 覆盖所有的子命令（cluster, node, certs, registry, download, iso, config）及各种参数（如 `--verbose`, `--yes`, `--config`, `--dry-run` 等）。
- **架构执行流**: 验证自顶向下的架构层级约束（Pipeline -> Module -> Task -> Step -> Runner -> Connector），确保不发生越级调用。
- **配置矩阵**: 测试不同 `kubernetes_type` (kubeadm/kubexm), `etcd.type` (kubeadm/kubexm/exists), 以及 `loadbalancer` 的各种部署排列组合。

## 3. 测试策略与方法
### 3.1 单元测试 (Unit Testing)
- **目标**: 验证内部通用工具包 (`internal/util`)、配置解析 (`config`)、日志 (`logger`) 以及基础执行器 (`runner`, `connector`) 的逻辑正确性。
- **方法**: 使用 Go 标准的 `testing` 库编写表驱动测试。对涉及文件系统交互的测试，必须使用 `t.TempDir()` 生成隔离环境。

### 3.2 接口与集成测试 (Integration Testing)
- **目标**: 确保 Task 和 Module 层能够正确编排 Step 的有向无环图 (DAG)，且 Runner 能正确下发指令。
- **方法**: 模拟本地 `local` 连接或配置模拟的 SSH 目标环境，验证 Pipeline 规划 (Plan) 阶段输出的 DAG 是否符合预期，执行阶段 (Run) 的状态传递是否正确。

### 3.3 端到端测试 (E2E Testing)
- **目标**: 从最终用户的视角验证业务场景。
- **环境要求**: 使用 KVM、VirtualBox 或云主机准备包含至少3个节点的测试集群环境。
- **核心场景**: 在线集群部署、离线包下载与内网完全离线部署、节点扩缩容、全链路备份与恢复。

### 3.4 幂等性与容错测试
- **幂等性**: 对相同的操作（如重复执行 `cluster create`），第二次执行必须全部 `Skip` 且不会引发集群状态的破坏。
- **原子性**: 模拟在某个 Step 执行中断（如手动 Kill 进程或断网），重新执行应能从上个检查点恢复。
- **离线测试**: 切断部署机的外网访问权限，仅利用 `kubexm download` 的产物进行全离线装机验证。

## 4. 架构规范检查
1. **严格分层检查**: Code Review 强制检查 Step 层是否直接调用了 Connector。必须通过 Runner 层。
2. **中心机器原则**: `host.yaml` 不允许出现 `127.0.0.1` 和 `localhost`，必须使用物理大网地址，确保完全的堡垒机视角。
3. **Panic 防护**: 所有的 Pipeline 必须包装在 `SafePlan` 中，不允许因模块代码的 Panic 导致整体工具崩溃。
