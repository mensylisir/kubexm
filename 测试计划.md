# Kubexm 全链路产品级测试计划 (v2.0 终极版)

## 一、 测试里程碑与时间轴

本测试计划分为四个相互依赖的里程碑，总工期预估 6 周。

### M1: 静态审计与单元质量保障 (Week 1)
- **范围**: `internal/util/`, `internal/cmd/`, `internal/config/`, `internal/connector/`。
- **任务**: 覆盖所有配置项读取校验、路径计算逻辑；完成所有 CLI 参数的补齐测试。保证底层 SSH Pool 并发锁定正确。
- **出口标准**: Unit Test 覆盖率 > 85%。静态代码检查零报错。

### M2: 执行图编排模拟 (Dry-Run 阶段) (Week 2)
- **范围**: `internal/pipeline/`, `internal/module/`, `internal/task/`。
- **任务**: 基于给定的所有 M01-M21 集群拓扑 YAML，注入 `--dry-run`，对比生成的图节点集合是否发生遗漏或循环依赖。重点测试在 Plan 阶段抛出特定异常，以防有副作用溢出到 Run 阶段。
- **出口标准**: 21 个拓扑的 Dry-Run 全部生成正确图快照且无副作用。

### M3: 核心分支 E2E 自动化演练 (Week 3-4)
- **范围**: 生命周期操作 `create`, `delete`, `upgrade`, `download` 离线全链条。
- **任务**: 在虚拟化靶场中真实部署这 21 套集群。执行 "搭建->离线资源载入->扩展节点->彻底销毁" 工作流。
- **出口标准**: K8s 组件全部 Running，且 Delete 操作后机器环境如初。

### M4: 高级组件及灾难恢复演练 (Week 5-6)
- **范围**: `certs`, `registry`, `backup`, `restore`, `reconfigure` 命令。
- **任务**: 破坏原有体系（如停用证书、停机），通过 restore 命令原地复活；模拟部分模块超时导致的长效回退操作。
- **出口标准**: 集群灾难恢复成功，证书热更无感。

## 三、 测试资源分配与风险规避

### 3.1 基础设施需求
- 资源池：至少提供 20+ vCPU 和 64GB+ 内存的 Hypervisor 用于 CI 流水线的并行靶场创建。
- 操作系统镜像要求：提前拉取 CentOS 7.9, Rocky 8, Ubuntu 22.04 离线 qcow2。

### 3.2 潜在风险点与兜底方案
1. **网络超时导致假阴性**: SSH 连接不稳定引发超时。
   - 对策: 测试脚本利用 Pipeline 参数或修改 Connector 强制拉长 `Timeout`，并启用 Retries 机制测试稳定性。
2. **状态残留引起连锁失败**: `cluster delete` 有缺陷导致下一次建群失败。
   - 对策: 自动化流水线每次执行必定采用完全全新实例或触发虚机快照回滚。
