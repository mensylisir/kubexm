# Kubexm 全面生命周期自动化测试计划 (v6.0 终极白皮书版)

## 一、 测试里程碑、范围与资源调配 (Gantt Chart & Phases)

该工程化计划预计持续 6~8 周，旨在通过完整的阶段划分，完成从静态层、执行逻辑层到底层虚拟化机群的全面自动化回归。

### 1.1 阶段一：解析器、核心连接层与架构护栏基线 (M1) (Week 1-2)
- **测试区域**: `cmd/` (CLI Cobra 解析器), `config/` (配置映射), `connector/` (SSH 池化), `util/`。
- **任务明细**:
  1. `Cobra.Command` 参数绑定与重载：校验 `--yes`, `--verbose`, `--dry-run`, `--skip-preflight`, `--force` 选项的短拼写和缺省拦截。所有包含交互式提示 (`y/N`) 的流程通过注入模拟输入来验证其自动化挂机模式是否有效。
  2. 反序列化异常检测：向 `host.yaml` / `config.yaml` 输入非法结构、空白或拼写错误的字段，验证系统 Defaulting 和强校验反馈。
  3. `LocalConnector` 与 `SSHConnector`：使用表驱动测试验证连接重试机制 (`Retries`)、超时控制 (`Timeout`)、上下文传递 (`Context`) 和安全密钥链认证 (`ssh-agent` / `id_rsa`)。
- **资源分配**: 开发者单机 `go test` 流水线。
- **出口准则**: Core 包 Unit Test 覆盖率达标 (>85%)，`golangci-lint` 无任何针对架构分层的越级调用警告。

### 1.2 阶段二：DAG 编排校验与异常控制流阻断 (M2) (Week 3)
- **测试区域**: `pipeline/`, `module/`, `task/`, `step/` 核心调度树。
- **任务明细**:
  1. **全矩阵图模拟**: 注入涵盖 `M01-M23` (23 个架构模型) 的虚拟配置文件，附带 `--dry-run` 指令运行。截获并反序列化引擎输出的 `ExecutionGraph`，利用算法检查环路与孤岛。
  2. 抛错拦截验证：对于特异分支（如试图执行尚未实现的 `UpgradeEtcdPipeline`）必须捕捉到对应的 `NotImplemented` 错误，而不是引发 Panic 破坏终端。
  3. 安全捕网验证：人为触发空指针，观察 `SafePlan` 包装器能否输出符合人类阅读的软性提示。
- **出口准则**: 所有 23 个拓扑的 Plan 生成规范、顺滑，控制流稳固。

### 1.3 阶段三：离线构建与靶机矩阵自动化验证 (M3) (Week 4-6)
- **测试区域**: 生命管理总线 (`create`, `delete`, `upgrade`, `download` 与离线探知)。
- **任务明细**:
  1. **[外网资源汲取]**: 在云端有公网 IP 的环境上，执行 `kubexm download --cluster=target`，收集 amd64/arm64 对应版本所需的全量二进制与容器镜像包。
  2. **[气隙/孤岛全自动装机]**: 把生成的 `packages/` 拷入断网实验靶机工作区。部署工具通过自动化感知到该离线目录，依序对 M01-M23 分别执行静默部署、服务探活 (`health -c all`)。
- **资源保障**: 由 Terraform、Packer 或 Ansible 支撑的 Hypervisor 测试平台 (ESXi/KVM)，保障各次部署前均使用一次性、纯净的临时虚拟机 (Ephemeral Environments)。
- **出口准则**: 23 套矩阵部署后，Kubernetes 核心控制面与 Worker 节点完全进入 `Running`，`kubectl get cs` 正常。随后的 `delete` 动作无残留，宿主网络栈与 `/etc/` 干净复原。

### 1.4 阶段四：高级灾难自愈、热重配与组件生命周期 (M4) (Week 7-8)
- **测试区域**: 扩缩容 `add-nodes`/`delete-nodes`, 证书轮换 `certs renew`, 私仓 `registry`, 配置快照 `backup` & `restore`, 以及热更新 `reconfigure` 命令。
- **任务明细**:
  1. **高可用扩展漂移**: 针对 `M16` 拓扑（含 External LB）集群动态添加和摘除 master/worker，观测 `keepalived` 的 VIP 转移和新节点的就绪时间。
  2. **证书热重签**: 测试 `certs renew` 子命令集（含 `kubernetes-ca`, `etcd-certs` 等），确保证书更新后节点双向认证持续互信，不引发服务雪崩。
  3. **抢救型数据恢复**: `backup` 创建环境快照 -> 直接 rm 掉 Master 节点的证书或者 kill ETCD -> `restore` 通过读取压缩快照原样恢复业务。
  4. **局部配置刷新**: `kubelet` 变更后运行 `reconfigure --restart` 验证单个守护进程的原子级更新。
- **出口准则**: 操作容灾必须在 15-30 分钟内完成；非法输入及跨界行为具备明确的错误提示反馈（如 snapshot path 缺漏）。

## 三、 测试前置条件与脏环境防御

### 3.1 基础设施及凭据安全机制
- **动态一次性秘钥 (Ephemeral SSH Keys)**: CI/CD 系统启动测试前自动生成 SSH Pair 下发至宿主，测试一旦结束或失败立刻抛弃秘钥。避免固定后门留存。
- **目标机器销毁原则**: 采用 Immutable Infrastructure (不可变基础设施) 模型，无论 `kubexm cluster delete` 能够成功将节点环境清理干净与否，都将在底层利用 Terraform 将该虚拟资产整机销毁。

### 3.2 对抗性混沌环境防雷策略 (Chaos Environments)
1. **磁盘残留与状态污染检测**:
   故意向一台预置用于 `cluster create` 的空靶机植入一个包含残余证书的 `/etc/kubernetes/` 或占满磁盘的 `/var/lib/etcd/` 目录。然后投递执行任务，验证 `PreflightModule` 预检阶段是否能精准捕捉到脏环境（“Directory not empty” 或 “Port in use”），并中断图流程，拒绝二次安装从而保护操作者现场。
2. **高危端口防冲突**:
   手动通过 `nc -l 6443` 或 `nc -l 2379` 占用 APIServer 或 ETCD 端口，强迫部署流程在预检步骤失败返回。
