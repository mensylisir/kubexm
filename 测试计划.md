# Kubexm 全链路产品级测试计划 (v3.0 终极详尽版)

## 一、 测试里程碑、范围与资源调配

本测试计划分为四个相互依赖的里程碑阶段，贯穿整个 `kubexm` 版本的生命周期，预估执行周期为 6-8 周，通过高度自动化的脚本及多环境验证。

### 1.1 M1 阶段：静态审计、单元质量与 CLI 解析 (Week 1-2)
- **目标**: 保障最底层的 `internal/util/`, `internal/cmd/`, `internal/config/`, `internal/connector/` 模块质量。
- **任务规划**:
  1. 覆盖所有 `cmd` 中包含 `--yes`, `--verbose`, `--dry-run` 以及各种组合选项如 `--skip-preflight`, `--force` 等参数的默认行为与传参校验。
  2. 针对 `host.yaml`, `config.yaml` 测试反序列化异常处理及边缘数据补偿（如默认 `arch` 回退为 `amd64`）。
  3. `LocalConnector` 和 `SSHConnector` 的长连接复用（Connection Pool）、超时（Timeout）、取消（Context Cancellation）隔离测试。
- **资源分配**: 静态单机 `go test ./...`。
- **出口准则**: Unit Test 覆盖率达标 (>85%)，`golangci-lint` 无严重警告。

### 1.2 M2 阶段：编排模拟、断点与异常控制流 (Week 3)
- **目标**: 保障 Pipeline 生成的 ExecutionGraph (DAG) 结构正确，依赖方向一致。
- **任务规划**:
  1. 以 `--dry-run` 模式，基于 23 种部署矩阵 (M01-M23) 的集群配置文件，断言引擎输出的执行图节点列表。
  2. 捕获非法图（例如 `UpgradeEtcdPipeline` 应报错）并拦截上抛异常，测试 `pipeline.SafePlan` 是否正确将 `panic` 转化为友好提示，而非导致 CLI 崩溃。
- **出口准则**: 所有合法配置均能稳定生成符合依赖树的 DAG；非法配置 100% 触发有效校验提示。

### 1.3 M3 阶段：离线环境构建与核心矩阵 E2E (Week 4-6)
- **目标**: 真实虚机环境部署 23 种拓扑结构，以及断网下载模式。
- **任务规划**:
  1. 联网机执行 `kubexm download --cluster=mycluster` 拉取。
  2. 纯内网靶场 (Air-gapped) 导入离线 `packages/`，按序部署 `M01-M23` 矩阵。
- **资源分配**: 基于 Terraform/Ansible 的 Hypervisor (如 ESXi/KVM 集群)，保障并发调度。
- **出口准则**: 23 套集群最终组件（Kube-APIServer, Kubelet, CoreDNS, Flannel/Calico）状态全变为 `Running`。

### 1.4 M4 阶段：高维灾备与全生命周期维护 (Week 7-8)
- **目标**: 覆盖证书更新、增删节点、备份与强力恢复操作。
- **任务规划**:
  1. 向 `M16` 拓扑集群动态添加/删除 worker 和 master (`add-nodes`, `delete-nodes`)，验证负载均衡漂移和同步能力。
  2. `certs renew` 子指令测试（`all`, `kubernetes-ca`, `etcd-certs` 等），确保证书更新后节点通信无阻。
  3. 通过 `backup` 打包凭据与快照，利用 `restore` 在遭受人为删库破坏后的原地恢复。
  4. 利用 `reconfigure` 动态变更 `kubelet` 并 `--restart`，确认只影响特定组件。
- **出口准则**: 所有状态变更与灾难自愈必须在 15 分钟内成功恢复，且无遗留脏数据。

## 三、 测试前置条件与风控防御机制

### 3.1 集群环境前置条件 (Pre-Requisites)
- 每套靶场必须保证通过 `ssh-copy-id` 确立了相互访问信任或具备密码环境，并在 `host.yaml` 精确描述。
- 每套靶场重置时间不能超过 5 分钟，依赖基础设施的 Snapshot 或者 Immutable Infrastructure 工具 (如 Packer)。

### 3.2 致命缺陷拦截机制与应急回滚
- **隔离污染风险**:
  所有在 `download` 与 `iso` 测试期间输出的文件，均指定到临时挂载卷 `/tmp/kubexm-artifacts/`。用例运行完毕，临时卷统一卸载，防止填满执行机的磁盘引发级联错误。
- **状态不一致阻断 (Dirty State Prevention)**:
  若 `cluster delete` 命令失效导致靶机残留 `/var/lib/etcd` 与 `/etc/kubernetes` 等敏感元数据，下一个 Pipeline 构建动作必须在预检时抛出 “集群非空状态异常” 以阻断新数据写入，保护现场环境供排查。
