# Kubexm 穷举型测试用例库 (全量指令集 & 边界条件 v4.0)

本文档致力于对整个 `kubexm` 树的每一个通过 Cobra 注册的子命令、参数变量（含缩写与互斥组合）、执行步骤及其底层的状态机变更，进行全面且结构化的验证。每条用例包含触发手段、API 动作与终态断言，以及严格遵循 `ARCHITECTURE_PLAN` 的架构审计。

---

## 1. 架构级防御与全局上下文用例 (Architecture Constraints & Global Flags)

| 用例 ID | 测试目标 | 执行命令组合 / 前置操作 | 断言预期与后台行为捕捉 | 核心架构检查点 |
|---|---|---|---|---|
| **AC-001** | 全局默认交互打断 (`--yes` 或 `-y`) | `kubexm cluster create -f config.yaml --yes` (在某未安装靶机) | 终端中没有任何交互式确认悬停 `(y/N)`。 | [ ] `PreflightModule` 无视用户键盘交互<br>[ ] 系统径直进入 DAG 引擎开始部署 |
| **AC-002** | 全局详细日志倾印 (`--verbose` 或 `-v`) | `kubexm --verbose version` | 能于 `stdout` 中抓取到底层 `logger.Debugf` 和追踪信息。 | [ ] 日志级别切为 Debug<br>[ ] 输出含有版本号、Git Hash 与 Commit 详情 |
| **AC-003** | 仿真干跑沙盒验证 (`--dry-run`) | 任意写操作 `kubexm cluster [cmd] --dry-run` | 屏幕输出按架构约束打印出的 `ExecutionGraph` 逻辑依赖树。**绝对禁止**进行目标机的真实写操作或服务拉起。 | [ ] 目标机器上 `/var/lib/` 毫无变化<br>[ ] DAG 节点打印的顺序完全符合规范 |
| **AC-004** | 中心堡垒机局域通信物理隔离 | 修改 `config.yaml` 或 `host.yaml` 塞入 `127.0.0.1` 节点后，执行 `create` | Plan 执行初期被直接拦截，日志抛出明确异常："Host validation failed: localhost not allowed." | [ ] 被首个验证器阻断<br>[ ] 确保系统强制使用网络级 SSH |
| **AC-005** | 引擎防 Panic 崩溃自愈测试 | 人为编译携带 `panic("test")` 代码块的开发测试包并执行任意指令 | 进程不挂起，也不打出大片混乱的 Golang Stacktrace。 | [ ] 被外层 `SafePlan` 包装器优雅捕获<br>[ ] 提示类似于 "pipeline plan failed: internal error" 的软终止信息 |

## 2. 核心集群全生命周期用例 (Cluster Operations)

### 2.1 Cluster Create (创建与节点初始化)
覆盖参数: `--config (-f)`, `--skip-preflight`, `--dry-run`, `--yes`

| 用例 ID | 测试目标与执行环境 | 执行命令 | 预期终态断言 |
|---|---|---|---|
| **CC-001** | 在线状态部署标准单/多主集群 (如 M01)。 | `cluster create -f M01.yaml` | 系统自动验证外部连接并顺利拉取依赖包，跑完全部组装 Step。登录被测机执行 `kubectl get componentstatuses` 均显示 OK。 |
| **CC-002** | 带外部专属负载均衡 (External LB) 创建 (如 M16)。 | `cluster create -f M16.yaml` | `LoadBalancerModule` 正确指配目标机组，`keepalived` 双活生效并且 VIP 能够发生故障漂移，代理后端的 APIServer 流量。 |
| **CC-003** | 纯净体系全二进制部署 (如 M02)。 | `cluster create -f M02.yaml` | 避开 Kubeadm 流程，由内置 Step 引擎生成全套 Systemd 服务并正确拉起 ETCD 与 APIServer，各个角色的 TLS 证书链完全建立双向互信。 |
| **CC-004** | **跳过预检指令**: 强制越过端口占用与内存容量预核查。 | `cluster create -f c.yaml --skip-preflight` | `PreflightModule` 被略过，引擎不管机子好坏强行下发安装 DAG，即使后续环境冲突失败也符合该参数的“强行下达”语义。 |
| **CC-005** | **离线感知模式**: 将断网靶场的工作区置入完整的 `packages/` 资源包。 | `cluster create -f c.yaml` | 工具扫描识别到该目录存在，触发**离线模式 (`OfflineMode=true`)**。通过自身的连接池使用 SSH/SFTP 分发 Tar 包并调用 Containerd 推入本地镜像，无需外网无一抛错。 |

### 2.2 Cluster Upgrade (集群版本升迁)
覆盖参数: `--config (-f)`, `--version (-t)`, `--dry-run`

| 用例 ID | 测试目标与执行环境 | 执行命令 | 预期终态断言 |
|---|---|---|---|
| **CU-001** | 滚动升级: 控制面和计算面平滑版本演进。 | `cluster upgrade -f c.yaml -t v1.26.0` | 系统先更新包和配置库，按照 `Drain -> Stop -> Patch -> Start -> Uncordon` 依次迭代 Master 与 Worker。业务探针不断联。 |
| **CU-002** | 负面用例: 缺失必填的升级目标版本标量。 | `cluster upgrade -f c.yaml` | 触发 CLI 拦截器，直接报错："target version (-t) is required for upgrade"。 |
| **CU-003** | 特别指令：剥离测试 ETCD 专属升级路径。 | `cluster upgrade-etcd -f c.yaml -t v3.5.9` | [开发断言] 遇到未完全实现的特性，抛出 "Etcd upgrade is not fully implemented"，并安全退出，不破坏当前库的执行逻辑树。 |

### 2.4 Cluster Backup & Restore (集群强力容灾机制)
覆盖参数: `--config (-f)`, `--type (-t)`, `--output (-o)`, `--backup (-b)`, `--snapshot (-s)`

| 用例 ID | 测试目标与执行环境 | 执行命令 | 预期终态断言 |
|---|---|---|---|
| **CBR-001** | 执行包含证书和数据库的全量灾备打点。 | `cluster backup -f c.yaml -t all -o /tmp/backups` | 扫描工作区与靶机并在 `/tmp/backups` 输出一份囊括全 PKI 证书、ETCD 快照与 Kubernetes Core 配置信息的存档包。 |
| **CBR-002** | 负面用例: 指向不被系统识别的灾备类型 (`-t wrong`)。 | `cluster backup -f c.yaml -t wrong` | 工具抛出明确警告或自动降级为 `all`，绝不产出半吊子脏文件。 |
| **CBR-003** | **毁天灭地级抢救**: 手动删掉 Master 控制面的核心文件或制造 ETCD 脑裂后，实施强制原位覆写还原。 | `cluster restore -f c.yaml -b /b.tar.gz -s /snap.db` | 集群核心组件被强制停服，原封不动刷入 `snap.db` 与 `b.tar.gz` 中的证书配置。重启拉起后可响应 `kubectl` 调度请求。 |
| **CBR-004** | 负面用例: 进行恢复时忘了挂载快照。 | `cluster restore -f c.yaml -b /b.tar.gz` | 工具于 Plan 阶段核验拦截："snapshot path is required for restore"。 |

### 2.5 Cluster Misc (节点群调节、排障与健康自查)
覆盖命令: `cluster delete`, `cluster scale`, `cluster health`, `cluster reconfigure`, `cluster manifests`
相关参数: `--force`, `--component (-c)`, `--restart`, `--backup`, `--new-config`

| 用例 ID | 测试目标与执行环境 | 执行命令 | 预期终态断言 |
|---|---|---|---|
| **CM-001** | 正常析构 (Teardown): 将运行中的集群环境完全清空销毁。 | `cluster delete -f c.yaml` | 系统沿 DAG 进行逆序回退卸载，解绑网络接口卡，清空 Iptables 和残留 Systemd 服务记录，机器回归洁净底座。 |
| **CM-002** | 强力拔除 (Teardown Force): 忽略部分资源驱逐挂起的软性约束。 | `cluster delete -f c.yaml --force` | `force` 指示令组件不管三七二十一，暴力拆毁进程与持久卷映射。 |
| **CM-003** | 健康遥测 (Heatlh Probing): 核查特定功能组件是否存活。 | `cluster health -c apiserver` | 连接对应节点检查其存活性，于标准输出得到 `Health: OK` 的确切信息。 |
| **CM-004** | 热重配 (Hot Reconfigure): 对单项组件打补丁及刷新。 | `cluster reconfigure -f old.yaml -n new.yaml -c kubelet --restart` | 解析并分发存在差异的 kubelet 参数模板，仅执行对应组件的 `systemctl restart`，其余服务纹丝不动。 |
| **CM-005** | 生成并旁路资源单 (Manifests Dump) [调试用途]。 | `cluster manifests -f c.yaml -o /tmp/man/` | 不进行底层应用，纯将配置对象化并输出静态 CNI/DNS 等资源的部署文件到本地。 |

## 3. Node Commands (单体计算节点调配与探针)

该部分深度覆盖调用 `client-go` SDK 产生的动作。
覆盖命令: `node list`, `node get`, `node cordon`, `node uncordon`, `node drain`
相关参数: `--cluster (-c)`, `--kubeconfig`, `--output (-o)`, `--grace-period`, `--force`, `--ignore-daemonsets`, `--timeout`

| 用例 ID | 测试目标与执行环境 | 执行命令 | 预期终态断言 |
|---|---|---|---|
| **NO-001** | 列出节点概览表 (List Nodes)。 | `node list -c mycluster --kubeconfig /k/conf` | 工具使用对应的身份证书请求 API Server，按表格打印节点的内部 IP、Ready 状态及 K8s 版本号。 |
| **NO-002** | 拉取特定节点详单并格式化 (Get Node YAML/JSON)。 | `node get worker-1 -c myc -o json` | 返回含元数据、Taints 污染与 Conditions 的完整 JSON 对象。 |
| **NO-003** | 阻塞计算资源池调度 (Cordon Node)。 | `node cordon worker-1 -c mycluster` | 修改 Spec 添加 `SchedulingDisabled` 标记，后续新的 Deployment 不再落脚此节点。 |
| **NO-004** | 解除节点调度封印 (Uncordon Node)。 | `node uncordon worker-1 -c mycluster` | 清除该对象的不可调度标记，将其重新接纳进算力池。 |
| **NO-005** | **终极排障疏散令** (Drain Node with Overrides)。 | `node drain worker-1 -c myc --force --ignore-daemonsets --grace-period=10` | 宽限 10 秒后强行将节点内无视约束的 Pod 清出宿主，确保节点变为空壳状态可拔电维护。 |

## 4. Certs Commands (安全证书生命周期与密钥轮换)

操作证书体系是导致生产雪崩的高危地带，此部分将做重点防呆校验。
覆盖命令: `certs renew`, `certs check-expiration`, `certs update`
子命令变体: `all`, `kubernetes-ca`, `kubernetes-certs`, `etcd-ca`, `etcd-certs`

| 用例 ID | 测试目标与执行环境 | 执行命令 | 预期终态断言 |
|---|---|---|---|
| **CE-001** | 全量重签安全通信凭据 (All Certs Rotate)。 | `certs renew all --yes` | Kubernetes 与 ETCD 的所有根 CA 及分支节点证书被更新并同步分发，触发对应依赖组件重启且新 TLS 无缝互信生效。 |
| **CE-002** | 专属翻新 K8s Root CA。 | `certs renew kubernetes-ca --yes` | **不更动 ETCD 的前提下**，仅重铸 Kubernetes CA 及相关从属叶证书，集群管控层洗牌完毕。 |
| **CE-003** | 专签业务级凭据 (Leaf Certs Rotate)。 | `certs renew kubernetes-certs --yes` | **不更动 Root CA 及证书链基础**，仅将到期将近的各终端交互证书有效日期延后，验证集群免遭信任链断裂停摆。 |
| **CE-004** | 集群日历审计 (Check Expiration)。 | `certs check-expiration` | 正确拉取各目录下的 PEM 文件，解析签发/到期界限时间戳，并在控制台提供预警天数倒计时。 |
