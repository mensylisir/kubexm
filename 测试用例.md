# Kubexm 全量指令与边界测试用例库 (v3.0 终极详尽版)

本文档是针对 `kubexm` 所有命令 (`Use:` 定义的指令) 及其所有的参数 (`Flags().` 定义的变体) 的穷举执行用例库，覆盖并遵循了所有前述的架构强约束 (例如中心化管理和 Step 隔离机制)。

---

## 1. 架构与边界防御用例 (Architecture Constraints)

| 用例 ID | 测试目标 | 执行命令与预置环境 | 断言预期与行为捕捉 | 检查项 (Checklist) |
|---|---|---|---|---|
| **AC-001** | 全局默认交互打断 | `kubexm cluster create -f config.yaml --yes` | 终端中没有任何交互式提示 `(y/N)`。 | [ ] 参数正确下发至所有 Module<br>[ ] 系统径直开始安装步骤 |
| **AC-002** | 全局提权与静默 | `kubexm --verbose version` | 能够抓取到所有执行器的 `logger.Debugf`。 | [ ] 输出内容显著增多<br>[ ] 打印了底层 API 版本与 commit 结构 |
| **AC-003** | `Dry-Run` 图校验与沙盒模式 | 任意 `kubexm [cmd] --dry-run` | 显示逻辑执行树且**绝对禁止**产生远端 I/O 操作及 `systemd` 改动。 | [ ] 目标机器文件时间戳未变<br>[ ] DAG 节点打印符合 `ARCHITECTURE_PLAN` 依赖序 |
| **AC-004** | 中心堡垒机物理大网隔离校验 | 在配置文件中塞入 `127.0.0.1` | Plan 阶段被直接拦截，日志打出："Host validation failed: localhost not allowed." | [ ] 在 `PreflightModule` 前期被强制终止 |
| **AC-005** | 代码级防 Panic 崩溃机制 | 人为编译携带空指针异常的测试包并执行 | 系统不应暴露庞杂的 Go 堆栈报错。 | [ ] 被 `SafePlan` 包装器优雅捕获<br>[ ] 提示 "pipeline plan failed: internal error" |

## 2. 集群生命周期全量用例 (Cluster Operations)

### 2.1 Cluster Create (创建集群)
覆盖命令: `cluster create`
参数: `--config (-f)`, `--skip-preflight`, `--dry-run`, `--yes`

| 用例 ID | 指令参数组合 | 测试重点与执行环境 | 预期终态断言 |
|---|---|---|---|
| **CC-001** | `cluster create -f M01.yaml` | 常规在线环境部署。验证 Kubeadm 基础逻辑。 | 系统智能拉取资源。部署完毕后执行 `kubectl get componentstatuses` 均显示 OK。 |
| **CC-002** | `cluster create -f M16.yaml` | 验证多主高可用与专属外部负载均衡编排。 | LB 节点的 `keepalived` 双活生效，VIP 能够互切并代理 APIServer 流量。 |
| **CC-003** | `cluster create -f M02.yaml` | 全二进制体系纯净部署。 | 不依靠 Kubeadm，通过直接配置和加载 Systemd 文件启动 ETCD 与 APIServer，并且 TLS 证书链完全互信可用。 |
| **CC-004** | `cluster create -f conf.yaml --skip-preflight` | 强制跳过软硬件与端口校验逻辑。 | 越过 `PreflightModule`，直接向节点下发部署流。 |
| **CC-005** | `cluster create -f conf.yaml` (离线自动感知) | 将包含全部容器和 Tar 归档的 `packages/` 目录置于工作区，且宿主机全部断网。 | 工具感知到离线模式开启，将通过 SSH 往各个节点进行纯本地化镜像推入和文件派送，无一外网异常。 |

### 2.2 Cluster Upgrade (集群与组件升级)
覆盖命令: `cluster upgrade`, `cluster upgrade-etcd`
参数: `--config (-f)`, `--version (-t)`, `--dry-run`, `--yes`

| 用例 ID | 指令参数组合 | 测试重点与执行环境 | 预期终态断言 |
|---|---|---|---|
| **CU-001** | `cluster upgrade -f conf.yaml -t v1.26.0` | 合法版本升级：验证滚动更新机制。 | 引擎按照 Drain->Stop->Patch->Start->Uncordon 流程依次更新 Master 和 Worker，业务 Pods 平滑驱逐，整体服务不断联。 |
| **CU-002** | `cluster upgrade -f conf.yaml` | 负面用例：不带目标版本号的无效执行。 | 工具拦截抛错，强制要求传入 `-t / --version` 参数。 |
| **CU-003** | `cluster upgrade-etcd -f conf.yaml -t v3.5.9` | [开发中] 测试异常上抛与隔离边界。 | 执行流被阻断并提示 "Etcd upgrade is not fully implemented"，不容许它破坏图逻辑。 |

### 2.3 Cluster Backup & Restore (备份与恢复)
覆盖命令: `cluster backup`, `cluster restore`
参数: `--config (-f)`, `--type (-t)`, `--output (-o)`, `--backup (-b)`, `--snapshot (-s)`

| 用例 ID | 指令参数组合 | 测试重点与执行环境 | 预期终态断言 |
|---|---|---|---|
| **CBR-001** | `cluster backup -f conf.yaml -t all -o /archive` | 执行集群各项配置与数据库文件的全量灾备打点。 | 在 `/archive` 下完整生成包含 PKI 树、ETCD 快照与 Kubernetes 关键配置的存档。 |
| **CBR-002** | `cluster backup -f conf.yaml -t invalid_type` | 负面用例：输入系统未支持的备份类型标识。 | 触发安全降级到默认类型 (`all`) 或明确报错阻断流程，禁止产生脏文件。 |
| **CBR-003** | `cluster restore -f conf.yaml -b /p.tar.gz -s /e.db` | 在主节点配置被破坏或 ETCD 脑裂时执行强力介入恢复。 | 系统停服并使用指定的快照覆盖受损数据，重启相关组件后 API Server 可正常调度。 |
| **CBR-004** | `cluster restore -f conf.yaml -b /p.tar.gz` | 负面用例：在恢复指令中缺失关键的数据库快照文件。 | 工具校验拦截："snapshot path is required for restore"。 |

### 2.4 Cluster Misc (重置、健康与销毁)
覆盖命令: `cluster delete`, `cluster health`, `cluster reconfigure`, `cluster scale`, `cluster manifests`
参数: `--force`, `--component (-c)`, `--restart`, `--backup`, `--new-config`

| 用例 ID | 指令参数组合 | 测试重点与执行环境 | 预期终态断言 |
|---|---|---|---|
| **CM-001** | `cluster delete -f conf.yaml` | 集群正常析构与释放。 | 反向回退部署图（如先清 Worker 后清 Master），清理残留配置文件与 Iptables 表。 |
| **CM-002** | `cluster delete -f conf.yaml --force` | 强力拔除节点群。 | 无视安全驱逐等待时间，暴力关闭 Systemd 服务与清理。 |
| **CM-003** | `cluster health -c apiserver` | 诊断探测命令验证。 | 通过内置指令或 Pod 探针校验 APIServer 的健康性并输出 `Health: OK`。 |
| **CM-004** | `cluster reconfigure -f conf.yaml -n new.yaml -c kubelet --restart` | 组件热变更逻辑。 | 检测并分发特定的 `kubelet` 变更配置并利用 Systemctl reload，其他组件不受任何扰动。 |
| **CM-005** | `cluster manifests -f conf.yaml -o /tmp/man` | [调试功能] 仅预演生成资源清单。 | 根据集群配置信息将 CoreDNS、CNI 的待执行 Yaml/Chart 提前吐出到本地，不上树。 |

## 3. 计算节点单体运维用例 (Node Commands)
覆盖命令: `node list`, `node get`, `node cordon`, `node uncordon`, `node drain`
参数: `--cluster (-c)`, `--kubeconfig`, `--output (-o)`, `--grace-period`, `--force`, `--ignore-daemonsets`, `--timeout`

| 用例 ID | 指令参数组合 | 测试重点与执行环境 | 预期终态断言 |
|---|---|---|---|
| **NO-001** | `node list -c mycluster --kubeconfig /my/kcf` | 解析证书并调取集群内的节点大名单。 | 表格打印集群内节点的状态（Ready/NotReady）、角色、版本号。 |
| **NO-002** | `node get worker-1 -c mycluster -o yaml` | 以格式化导出特定节点的元数据。 | 精准返回指定节点的完整 Spec 和 Status 详情（含 Taints 与 Conditions）。 |
| **NO-003** | `node cordon worker-1 -c mycluster` | 将计算节点置为禁止进入状态。 | 给对象打上 `SchedulingDisabled` 标记，拒绝新 Pod 分配。 |
| **NO-004** | `node uncordon worker-1 -c mycluster` | 撤销节点的封锁指令。 | 消除 `SchedulingDisabled` 标记，恢复算力池负载。 |
| **NO-005** | `node drain worker-1 -c mycluster --force --ignore-daemonsets --grace-period=10` | 暴力排空节点上的负载并安全下线。 | 无视 DaemonSet 报警，无视单体孤儿 Pod 的约束，在 10s 的宽限期后强行将节点腾空供维护使用。 |

## 4. 证书系统轮转与核查用例 (Certs Commands)
覆盖命令: `certs renew`, `certs check-expiration`, `certs update`
参数: `--yes`

| 用例 ID | 指令参数组合 | 测试重点与执行环境 | 预期终态断言 |
|---|---|---|---|
| **CE-001** | `certs renew all --yes` | 全量更新安全通信凭据。 | Kubernetes 与 ETCD 的根 CA 和子级叶证书悉数完成重签并发放到相关机位，各组件重加载无缝。 |
| **CE-002** | `certs renew kubernetes-ca --yes` | 专对 K8s CA 证书实施轮换。 | K8s 控制面内的所有证书根据新的根密钥链重签，而 ETCD 通道证书原样保持。 |
| **CE-003** | `certs renew etcd-certs --yes` | 保留根的前提下静默更新叶节点。 | 不更动信任链（CA），只替各 ETCD Peer 颁发有效期更远的新证。 |
| **CE-004** | `certs check-expiration` | 核算集群内的过期日历。 | 扫描全目录，精准解析出各项证书（如 `apiserver.crt`）的倒计时剩余天数或过期标识。 |
