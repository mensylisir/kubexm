# Kubexm 全量测试用例库

本文档为执行手册级别测试用例库。每一条测试用例包含了具体的预置条件、执行步骤、及检验标准。

## 1. 基础架构及边界条件测试 (Architecture Constraints)

### TC-AR-001: 强制本地堡垒机大网通信原则
- **前置条件**: 一台测试机，其大网 IP 为 `192.168.1.100`。
- **执行步骤**:
  1. 编写配置文件 `conf.yaml`，将 target host 的 IP 地址设为 `127.0.0.1` 或 `localhost`。
  2. 执行 `kubexm cluster create -f conf.yaml`。
- **期望结果**:
  - `PreflightModule` (或对应的预检 Step) 直接拒绝执行。
  - 抛出明确的 Error 日志："Host mapping prohibits loopback addresses. Please use external network IP."
  - 任务非正常中断。

### TC-AR-002: Pipeline Panic 防护与上抛机制
- **前置条件**: 编译一个带有针对性 Bug (在某 Task Plan 中触发 `panic("simulated")`) 的 kubexm 二进制版本。
- **执行步骤**:
  1. 运行包含该 Bug 分支的 Pipeline 操作。
- **期望结果**:
  - CLI 工具不应出现未捕获的 Panic 从而输出长篇的 golang stacktrace。
  - `pipeline.SafePlan()` 成功拦截 Panic 并转换为 Go error 上抛。
  - 用户看到友好的终止提示 "Pipeline planning failed due to internal error."

### TC-AR-003: Step 幂等性保障验证
- **前置条件**: 一个由 `kubexm` 成功部署且健康的 `M01` 集群。
- **执行步骤**:
  1. 在没有任何变动的情况下，重新执行 `kubexm cluster create -f conf.yaml`。
- **期望结果**:
  - 整个执行流迅速跑完。
  - 终端中各 Step (例如 `InstallContainerdStep`, `GenerateCertsStep`) 的状态为 `[Skip]`。
  - 安装前后集群服务没有任何重启，文件修改时间戳未变。

## 2. 全局参数及杂项功能测试

### TC-GL-001: 全局 Yes 参数传递
- **执行步骤**: 执行 `kubexm cluster delete -f conf.yaml --yes`。
- **期望结果**: 无任何 "(y/N)" 的控制台停顿，系统径直执行高危操作。

### TC-GL-002: Dry-run 规划与图校验
- **执行步骤**: 对任意 M01-M21 架构执行 `kubexm cluster create -f conf.yaml --dry-run`。
- **期望结果**:
  - 终端输出经过规划生成的任务依赖树。
  - 不会向远端发起任何文件写入和进程执行指令。

## 3. 集群生命周期用例 (Cluster Commands)

### TC-CL-001: 完全离线环境下的集群创建
- **前置条件**:
  - 一台有外网的跳板机 A，一台无外网的部署机 B (和一组空节点 C, D)。
  - `conf.yaml` 配置妥当。
- **执行步骤**:
  1. 在机 A 执行 `kubexm download -f conf.yaml -o packages/`。
  2. 验证机 A 是否跳过了 Host 可达性检查并拉取了全部 `tar.gz` 及 `images` 归档。
  3. 将 `packages/` 目录和二进制拷贝到机 B 工作区。
  4. 机 B 上拔除公网网卡，执行 `kubexm cluster create -f conf.yaml`。
- **期望结果**:
  - 系统智能探测到离线模式 (`OfflineMode=true`)。
  - Connector 从本地 `packages/` 目录通过 SSH SCp/SFTP 将文件全分发给空节点。
  - 集群在离线状态下部署成功，各 Node 及 System Pods 变为 `Running`。

### TC-CL-002: 集群平滑扩容 (Scale Out)
- **前置条件**: 一个正常运行的 M01 单主节点集群。
- **执行步骤**:
  1. 修改 `conf.yaml`，在 roles 中向两台新空机器分配 `worker` 角色。
  2. 运行扩容相关命令（或直接运行 create 命令触发增量调谐）。
- **期望结果**:
  - 仅对新增节点运行系统初始化、容器运行时安装、Certs 颁发和 Kubelet 加入操作。
  - 原有主节点不受影响，新 Worker 在 `kubectl get nodes` 展现为 `Ready`。

### TC-CL-003: 集群整体版本升级
- **前置条件**: 集群运行 K8s 版本 `v1.24.0`，所有节点 Ready。
- **执行步骤**:
  1. 执行 `kubexm cluster upgrade -f conf.yaml -t v1.25.0`。
- **期望结果**:
  - Master 节点通过滚动方式（Drain -> Stop -> Upgrade Binaries/Configs -> Start -> Uncordon）升级。
  - 接着 Worker 节点依次升级。
  - 整个过程中被测应用不能全盘中断。集群最终版本更新为 `v1.25.0`。

### TC-CL-004: 核心组件备份与完全恢复
- **前置条件**: 一个存有业务 Pod 和相关配置的健康集群。
- **执行步骤**:
  1. 运行 `kubexm cluster backup -f conf.yaml -t all -o /tmp/backup` 生成备份产物。
  2. 手动去远端删除 K8s CA 证书，并停止 ETCD，破坏集群。
  3. 运行 `kubexm cluster restore -f conf.yaml -b /tmp/backup/full.tar.gz -s /tmp/backup/snapshot.db`。
- **期望结果**:
  - PKI 证书恢复到原始路径并权限正确设置。
  - ETCD 根据 Snapshot 重新启动。
  - 业务和集群配置全部恢复为执行备份前的状态。

## 4. 节点日常管理用例 (Node Commands)

### TC-NO-001: 获取节点状态详情
- **执行步骤**: 执行 `kubexm node get node01 -c mycluster -o yaml`。
- **期望结果**: 工具使用内置加载的 Kubeconfig，调用 Kubernetes API Server，并格式化输出 node01 的详细 Yaml，其内容包含 Taints, Allocatable Resources 等信息。

### TC-NO-002: 强力排空异常节点
- **前置条件**: `worker-02` 上跑着由 DaemonSet 和独立的无控制器管理的 Pod。
- **执行步骤**: `kubexm node drain worker-02 -c mycluster --force --ignore-daemonsets`。
- **期望结果**:
  - 节点变更为不可调度状态 (`Cordoned`)。
  - `force` 标记生效，无控制器的孤儿 Pod 亦被安全清理，最终腾空该节点。

## 5. 证书与外围资产管理用例 (Certs & Registry)

### TC-CE-001: 细粒度更新 Kubernetes Leaf 证书
- **前置条件**: 集群正常运行。
- **执行步骤**: `kubexm certs renew kubernetes-certs --yes`。
- **期望结果**:
  - 不触碰 Root CA (`ca.crt`/`ca.key`) 及其信任链。
  - 仅重新生成类似 `apiserver.crt`、`apiserver-kubelet-client.crt` 这样的端点证书。
  - 重启受影响的静态 Pod 或 Systemd 进程使其加载新证书，检查系统仍然连通。

### TC-UT-001: 动态制作自给自足的 ISO 安装包
- **前置条件**: 环境装有 `xorriso` 等构建工具。
- **执行步骤**: 执行 `kubexm iso create -f conf.yaml -o kubexm-offline.iso`。
- **期望结果**:
  - 执行引擎依照 `templates/iso` 的指令编译生成。
  - 工具完成 Grub 及 ISOLINUX 的引导修改。
  - 最终的 ISO 体积可观，并包含完整的 OS 依赖库及对应架构的 Kubernetes RPM/DEB/Binaries。
