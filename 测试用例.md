# Kubexm 全量指令与边界测试用例库 (v5.0 终极白皮书版)

本文档致力于对整个 `kubexm` 树的每一个通过 Cobra 注册的子命令、参数变量（含缩写与互斥组合）、执行交互及其底层的状态机变更，进行最极端的验证。每条用例包含触发命令、断言预期、依赖库和 API 等检查点。

---

## 1. 架构级防御与全局交互环境用例 (Global Context & Architecture Constraints)

| 用例 ID | 测试目标 | 执行命令组合 / 前置操作 | 断言预期与后台行为捕捉 | 核心架构检查点 |
|---|---|---|---|---|
| **AC-001** | 全局默认交互打断 (`--yes` 或 `-y`) | `kubexm cluster delete -f config.yaml --yes` | 终端中没有任何交互式确认悬停 `(y/N)`。 | [ ] `PreflightModule` 无视用户键盘交互<br>[ ] 系统径直进入 DAG 引擎开始高危操作 |
| **AC-002** | 全局详细日志倾印 (`--verbose` 或 `-v`) | `kubexm --verbose version` | 能于 `stdout` 中抓取到底层 `logger.Debugf`。 | [ ] 日志级别切为 Debug<br>[ ] 打印包含 Git Hash 的结构体追踪信息 |
| **AC-003** | 仿真干跑沙盒验证 (`--dry-run`) | 任意写操作 `kubexm cluster [cmd] --dry-run` | 输出按架构约束编排出的 `ExecutionGraph` 逻辑依赖树。**绝对禁止**进行目标机的真实写操作。 | [ ] 目标机器上文件时间戳未变<br>[ ] DAG 节点顺序符合 `ARCHITECTURE_PLAN` |
| **AC-004** | 中心堡垒机物理局域网隔离校验 | 在配置文件中塞入 `127.0.0.1` 节点后，执行 `create` | Plan 执行初期被直接拦截，日志抛出："Host validation failed: localhost not allowed." | [ ] 在 `PreflightModule` 前期被强制终止<br>[ ] 系统强制拒绝环回地址 |
| **AC-005** | 防 Panic 崩溃自愈与越权访问检查 | 利用携带 `panic("test")` 开发包执行；人工扫代码。 | 系统不暴露混乱的 Golang Stacktrace。所有 `Step` 均不直接调用 `Connector` API。 | [ ] 被 `SafePlan` 包装器优雅捕获<br>[ ] 提示 "pipeline plan failed: internal error" |

## 2. 集群全生命周期用例 (Cluster Operations)

### 2.1 Cluster Create (创建与节点初始化)
覆盖参数: `--config (-f)`, `--skip-preflight`, `--dry-run`, `--yes`

| 用例 ID | 指令参数组合 | 测试重点与执行环境 | 预期终态断言 |
|---|---|---|---|
| **CC-001** | `cluster create -f M01.yaml` | 在线状态部署标准单/多主集群 (如 M01)。 | 系统自动验证外部连接并顺利拉取依赖包，跑完全部组装 Step。K8s 控制面各组件 `Running`。 |
| **CC-002** | `cluster create -f M16.yaml` | 带外部专属负载均衡 (External LB) 创建 (如 M16)。 | `LoadBalancerModule` 编排 `keepalived` 双活生效，VIP 能够故障漂移，并代理 APIServer 流量。 |
| **CC-003** | `cluster create -f M02.yaml` | 纯净体系全二进制部署 (如 M02)。 | 不依靠 Kubeadm，通过直接配置和加载 Systemd 服务并正确拉起 ETCD 与 APIServer，TLS 证书全链双向互信可用。 |
| **CC-004** | `cluster create -f c.yaml --skip-preflight` | 强制跳过软硬件与端口校验逻辑。 | 越过 `PreflightModule` 探测阶段，引擎不管机子是否存在端口冲突强行下发安装 DAG 流。 |
| **CC-005** | `cluster create -f c.yaml` (离线自动感知) | 将包含全部容器镜像和二进制 Tar 归档的 `packages/` 目录置于工作区，宿主机全部断网。 | 工具感知到离线模式 (`OfflineMode=true`) 开启。引擎利用自身 SSH 连接池向各个节点进行镜像推入和包派送，部署全程不发一笔外网请求。 |

### 2.2 Cluster Upgrade (集群与组件升级)
覆盖命令: `cluster upgrade`, `cluster upgrade-etcd`
参数: `--config (-f)`, `--version (-t)`, `--dry-run`, `--yes`

| 用例 ID | 指令参数组合 | 测试重点与执行环境 | 预期终态断言 |
|---|---|---|---|
| **CU-001** | `cluster upgrade -f c.yaml -t v1.26.0` | 合法版本升级：验证滚动更新机制。 | 引擎按照 `Drain -> Stop -> Patch -> Start -> Uncordon` 依次迭代 Master 与 Worker。集群内业务 Pods 平滑驱逐，整体服务接续可用。 |
| **CU-002** | `cluster upgrade -f c.yaml` | 负面用例：缺失必填的升级目标版本标量。 | 触发 CLI 拦截器，直接报错终止："target version (-t) is required for upgrade"。 |
| **CU-003** | `cluster upgrade-etcd -f c.yaml -t v3.5.9` | [预设防护] 独立的 ETCD 升级执行路径。 | 遇到未完全实现的特性，抛出 "Etcd upgrade is not fully implemented" 安全退出，不破坏当前库的执行逻辑树。 |

### 2.3 Cluster Backup & Restore (集群强力灾备与复苏)
覆盖参数: `--config (-f)`, `--type (-t)`, `--output (-o)`, `--backup (-b)`, `--snapshot (-s)`

| 用例 ID | 指令参数组合 | 测试重点与执行环境 | 预期终态断言 |
|---|---|---|---|
| **CBR-001** | `cluster backup -f c.yaml -t all -o /tmp/backups` | 执行包含证书和数据库的全量灾备打点。 | 扫描工作区与靶机，在指定目录输出包含全 PKI 证书、ETCD 快照与 Kubernetes Core 配置的 Tar 归档。 |
| **CBR-002** | `cluster backup -f c.yaml -t wrong` | 负面用例：输入系统未支持的备份类型标识。 | 触发安全降级到默认类型 (`all`) 或明确报错阻断流程，禁止产生脏文件。 |
| **CBR-003** | `cluster restore -f c.yaml -b /b.tar.gz -s /snap.db` | **毁天灭地级抢救**: 删除 Master 核心文件或制造 ETCD 脑裂后，实施强制原位覆写还原。 | 系统停服并使用指定的快照覆盖受损数据。重启相关组件后 API Server 数据全部恢复为打快照那一刻的状态。 |
| **CBR-004** | `cluster restore -f c.yaml -b /b.tar.gz` | 负面用例：在恢复指令中缺失关键的数据库快照文件。 | 工具于 Plan 阶段核验拦截并阻断执行："snapshot path is required for restore"。 |

### 2.4 Cluster Misc (重置析构、探针、热重配与清单抛出)
覆盖命令: `cluster delete`, `cluster health`, `cluster reconfigure`, `cluster scale`, `cluster manifests`
参数: `--force`, `--component (-c)`, `--restart`, `--backup`, `--new-config`

| 用例 ID | 指令参数组合 | 测试重点与执行环境 | 预期终态断言 |
|---|---|---|---|
| **CM-001** | `cluster delete -f c.yaml` | 正常析构 (Teardown): 将运行中的集群环境完全清空销毁。 | 系统沿 DAG 逆序回退，解绑网络接口卡，清空 Iptables 和残留 Systemd 服务记录，机器回归部署前的原初洁净底座。 |
| **CM-002** | `cluster delete -f c.yaml --force` | 强力拔除 (Teardown Force): 忽略部分资源驱逐超时的约束。 | 无视安全驱逐等待时间，暴力关闭 Systemd 服务与持久卷映射并清理。 |
| **CM-003** | `cluster health -c apiserver` | 诊断探测 (Health Probing): 核查特定功能组件是否存活。 | 连接对应节点检查其存活性，于标准输出精准返回 `Health: OK`。 |
| **CM-004** | `cluster reconfigure -f old.yaml -n new.yaml -c kubelet --restart` | 组件热变更 (Hot Reconfigure): 对单项组件打补丁。 | 引擎计算差异，向各个节点分发新的 `kubelet` 参数模板，并触发 `systemctl restart`，其余组件不受任何扰动。 |
| **CM-005** | `cluster manifests -f c.yaml -o /tmp/man/` | 资源清单脱水 (Manifests Dump) [调试预演]。 | 不进行底层部署应用，纯将配置对象化并输出静态 CNI/DNS 等资源的 yaml 部署文件到本地 `/tmp` 目录供人工审阅。 |

## 3. Node Commands (单体计算节点调配与探针)

该模块主要测试对于 K8s API (Client-Go) 的直接交互包装能力。
覆盖命令: `node list`, `node get`, `node cordon`, `node uncordon`, `node drain`
相关参数: `--cluster (-c)`, `--kubeconfig`, `--output (-o)`, `--grace-period`, `--force`, `--ignore-daemonsets`, `--timeout`

| 用例 ID | 指令参数组合 | 测试重点与执行环境 | 预期终态断言 |
|---|---|---|---|
| **NO-001** | `node list -c mycluster --kubeconfig /my/kcf` | 解析独立证书调取集群内的节点大名单。 | 控制台表格排版打印出集群内全部节点的内部 IP、Ready 状态及 K8s 版本。 |
| **NO-002** | `node get worker-1 -c myc -o json` | 拉取特定节点详单并强制 JSON 格式化输出。 | 返回含完整对象元数据、Taints (污点) 与 Conditions 的 JSON 字符串。 |
| **NO-003** | `node cordon worker-1 -c myc` | 将计算节点置为禁止进入调度状态。 | 向该节点的 Spec 对象打上 `SchedulingDisabled` 标记，后续新的 Deployment Pod 不再落脚此节点。 |
| **NO-004** | `node uncordon worker-1 -c myc` | 解除节点的不可调度封印指令。 | 消除节点上的 `SchedulingDisabled` 标记，恢复算力池常规调度能力。 |
| **NO-005** | `node drain worker-1 -c myc --force --ignore-daemonsets --grace-period=10` | 暴力排空节点上的业务负载并安全下线 (配合各类越权覆写符)。 | 无视 DaemonSet 报警，无视单体孤儿 Pod 的约束，并在明确的宽限期后强行将节点腾空供换件维护使用。 |

## 4. Certs Commands (安全证书生命周期与密钥轮换)

操作 TLS 证书体系是导致生产环境雪崩的高危地带，此部分将做重点级幂等断言。
覆盖命令: `certs renew`, `certs check-expiration`, `certs update`
子命令变体: `all`, `kubernetes-ca`, `kubernetes-certs`, `etcd-ca`, `etcd-certs`

| 用例 ID | 指令参数组合 | 测试重点与执行环境 | 预期终态断言 |
|---|---|---|---|
| **CE-001** | `certs renew all --yes` | 全量更新所有系统级安全通信凭据。 | Kubernetes 与 ETCD 的所有根 CA 及分支节点证书被更新重签并全域同步分发，触发依赖组件重载且新 TLS 证书双向握手互信成功。 |
| **CE-002** | `certs renew kubernetes-ca --yes` | 专属翻新 Kubernetes 侧的 Root CA。 | 不更动 ETCD 的前提下，重铸 K8s CA 并利用该新 CA 下游重签所有的 K8s 叶证书，集群管控层全面洗牌完成。 |
| **CE-003** | `certs renew kubernetes-certs --yes` | 仅续签到期的业务级凭据 (Leaf Certs Rotate)。 | 不更动核心 Root CA 及根证书信任链，仅将到期将近的各控制终端与代理节点的叶级证书（如 `apiserver.crt`）有效日期延后，验证集群免遭信任链断裂停摆。 |
| **CE-004** | `certs check-expiration` | 全局核查证书集群过期审计。 | 工具扫描所有节点的各个认证目录下的 PEM 文件，解析证书内部数据戳，并在控制台以天数预警显示倒计时。 |
