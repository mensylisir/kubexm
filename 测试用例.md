# Kubexm 全景式质量保障与用例图谱 (v6.0 终极全量版)

本文档将 Cobra 命令行引擎 (`internal/cmd/`) 注册的所有的指令集 (Use)、缩写及长参数变体 (`Flags().`) 与后台架构强制标准相连结，构建出极为严苛的穷举性回归库。每一条款严格包含前置条件、行为步骤、终态断言，甚至是底层代码（如 Pipeline）的流向断言。

## 1. 架构级防御、全局交互及上下文环境约束 (Architecture Constraints)

| 用例 ID | 测试靶点 / 对应参数 | 执行环境与输入指令 | 断言预期与行为捕捉 | 核心架构检查点 (CI/CD 护栏) |
|---|---|---|---|---|
| **AC-001** | 全局交互打断 / 自动化挂机 (`--yes` 或 `-y`) | `kubexm cluster delete -f config.yaml --yes` (对于存量集群的高危操作) | 终端中绝对不能悬停出现 `(y/N)` 提示符，直接向系统下达销毁令。 | [ ] `PreflightModule` 完全屏蔽标准输入<br>[ ] 系统立即跨入 `ExecutionGraph` 引擎 |
| **AC-002** | 全局 Debug 倾印 (`--verbose` 或 `-v`) | `kubexm --verbose version` 或随附任意指令 | 能在 `stdout` 中抓取到底层组件的 `logger.Debugf` 和 API 握手过程追踪信息。 | [ ] 日志输出大幅增量<br>[ ] 输出含有 Git Hash, 平台标识等底座详情 |
| **AC-003** | 仿真图校验与沙盒干跑 (`--dry-run`) | 发起修改动作 `kubexm cluster [cmd] --dry-run` | 控制台按照规范渲染出所选部署架构对应的 `ExecutionGraph` 逻辑依赖树。**严格禁止**任何远端 I/O 和 `systemd` 改写。 | [ ] 目标机器文件时间戳、进程列表未产生任何异动<br>[ ] 节点编排序列符合 `ARCHITECTURE_PLAN` 的无环有向树 |
| **AC-004** | 物理环回网卡拦截 (Bastion Host 校验) | 把 `host.yaml` 中的管理目标 IP 修改为 `127.0.0.1` 或是解析至本机的域名，执行 `create` | 在刚出入栈的 `Plan` 初期被直接击落，日志必定抛出："Host validation failed: localhost not allowed." | [ ] 在 `PreflightModule` 第一步即强制终止<br>[ ] 引擎绝对拒绝与环回口通信 |
| **AC-005** | 防 Panic 级自愈与越层检测 | [研发测试包] 利用携带 `panic("mock")` 的 `Step` 包执行。 | 系统绝不把冗长且混乱的 Golang Stacktrace 推给用户。此外，审计扫描 `internal/step/` 目录不许直接关联 `internal/connector`。 | [ ] 由顶层 `SafePlan` 包装器优雅捕获<br>[ ] 界面上呈现 "pipeline plan failed: internal error" 软性终止信息 |

## 2. 核心集群全生命周期用例 (Cluster Operations)

涵盖核心生命周期管理，涉及到的指令包含 `create`, `delete`, `upgrade`, `upgrade-etcd`, `backup`, `restore`, `health`, `scale`, `reconfigure`, `manifests` 等。

### 2.1 Cluster Create (装机及初始化)
包含参数: `--config (-f)`, `--skip-preflight`, `--dry-run`, `--yes`

| 用例 ID | 组合指令输入 | 测试意图与执行环境 | 终态断言与校验标准 |
|---|---|---|---|
| **CC-001** | `cluster create -f M01.yaml` | 验证公网连通时的**标准在线部署** (如原生 Kubeadm 模型 M01)。 | 工具智能拉取依赖，执行所有 `Task` 编排。登录集群执行 `kubectl get componentstatuses` 均处于 `Healthy/OK`。 |
| **CC-002** | `cluster create -f M16.yaml` | 验证多主高可用外挂负载均衡池 (External LB, M16) 方案。 | 在专属 LB 节点上的 `keepalived` 双活完全生效并持有可用 VIP；且可通过该 VIP 连通并代理至后端的各 Master APIServer。 |
| **CC-003** | `cluster create -f M02.yaml` | 验证去 Kubeadm 依赖的**纯血二进制体系**部署 (M02)。 | 通过引擎自动生成的 Systemd 服务文件直接拉起 ETCD 与 APIServer；并验证由工具颁发给各角色（Node, Member）的 TLS 证书全链双向互信可用。 |
| **CC-004** | `cluster create -f c.yaml --skip-preflight` | 强制压制软硬件连通性与预检。 | 引擎直接跳过 `PreflightModule` 的空间和端口检查步骤，不管被测机是否存在端口 6443 冲突也强行下发安装 DAG 图（即使后期报错）。 |
| **CC-005** | `cluster create -f c.yaml` (无网离线靶场) | 把预打包了各类容器镜像和编译 Tar 的 `packages/` 归档置入工作区，并**掐断**外部路由器网络。 | 工具智能检测到离线状态 (`OfflineMode=true`)。基于内置的 SSH/SFTP 池把本地缓存内的包逐一派送推向目标节点，安装从头至尾无外网依赖、无 HTTP Timeout 抛出。 |

### 2.2 Cluster Upgrade (平台级升迁)
包含指令: `cluster upgrade`, `cluster upgrade-etcd`
包含参数: `--config (-f)`, `--version (-t)`, `--dry-run`, `--yes`

| 用例 ID | 组合指令输入 | 测试意图与执行环境 | 终态断言与校验标准 |
|---|---|---|---|
| **CU-001** | `cluster upgrade -f c.yaml -t v1.26.0` | 合规升级：系统无缝滚动跨版本。 | 工具先完成资源替换，严格按 `Drain -> Stop -> Patch -> Start -> Uncordon` 步骤逐台更替 Master 和 Worker。在此期间集群应用探针不会全面中断，最终 `kubectl version` 返回 v1.26.0。 |
| **CU-002** | `cluster upgrade -f c.yaml` | 负向用例：遗漏系统强制的版本入参。 | 指令执行被 Cobra 拦截器掐断并报："target version (-t) is required for upgrade"，防止盲目操作。 |
| **CU-003** | `cluster upgrade-etcd -f c.yaml -t v3.5.9` | [预防性挂起] 阻断未完善功能的越级侵入。 | 系统探测到目标未实现，安全弹出 "Etcd upgrade is not fully implemented" 警告终止流程，而不影响现有的执行图库。 |

### 2.3 Cluster Backup & Restore (核心数据抢救与回滚)
包含指令: `cluster backup`, `cluster restore`
包含参数: `--config (-f)`, `--type (-t)`, `--output (-o)`, `--backup (-b)`, `--snapshot (-s)`

| 用例 ID | 组合指令输入 | 测试意图与执行环境 | 终态断言与校验标准 |
|---|---|---|---|
| **CBR-001** | `cluster backup -f c.yaml -t all -o /bks` | 导出涵盖了证书密钥树、数据库记录的全量快照存档。 | 工作区下正确输出包含了全套 PKI 结构树、ETCD 快照与各项 Kubernetes 组件 Config 的归档包。 |
| **CBR-002** | `cluster backup -f c.yaml -t junk` | 负向用例：输入无法识别的备份类型。 | 抛出不支持类型警告或者回退到安全全集类型 (`all`)，不产出不完整或结构乱序的脏备份。 |
| **CBR-003** | `cluster restore -f c.yaml -b /p.tar.gz -s /e.db` | **核级毁灭复原**: RM 掉控制面证书树并致使 ETCD 服务崩溃，随后强行执行该命令覆盖现场。 | 工具停止存留服务，精准剥离还原证书，并将 `e.db` 的数据注回 ETCD。集群重启后响应状态如快照打点时一致，K8s 各组件全部复活。 |
| **CBR-004** | `cluster restore -f c.yaml -b /p.tar.gz` | 负向用例：强力恢复时忘了传递必要的快照数据链。 | Plan 层核验失败并报错："snapshot path is required for restore"。 |

### 2.4 Cluster Misc (重置析构、探针及热重载清单)
包含指令: `cluster delete`, `cluster scale`, `cluster health`, `cluster reconfigure`, `cluster manifests`
相关参数: `--force`, `--component (-c)`, `--restart`, `--backup`, `--new-config`

| 用例 ID | 组合指令输入 | 测试意图与执行环境 | 终态断言与校验标准 |
|---|---|---|---|
| **CM-001** | `cluster delete -f c.yaml` | 干净的析构动作 (Teardown): 完全退场清理靶机。 | 引擎生成反向回滚图，解绑各虚拟网络层，清理遗留 Iptables、卸载 Systemd 进程和数据卷。主机恢复白板状态。 |
| **CM-002** | `cluster delete -f c.yaml --force` | 暴力析构 (Teardown Force): 无视节点清空的等待。 | 跳过资源驱逐等待期的软束缚，采用 `SIGKILL` 级手段闭毁服务与解挂分区点。 |
| **CM-003** | `cluster health -c apiserver` | 遥测探测 (Health Probing): 监控特定进程。 | 发送连通或健康 HTTP 请求，成功收回判定并在屏幕印出 `Health: OK`。 |
| **CM-004** | `cluster reconfigure -f old.yaml -n new.yaml -c kubelet --restart` | 组件热变更 (Hot Reconfigure): 细粒度的差异补丁。 | 读取出 `new.yaml` 中关于 kubelet 专属的改动差集分发下去，调用 Systemctl reload 操作特定进程，其他组件静默运转。 |
| **CM-005** | `cluster manifests -f c.yaml -o /tmp/` | 清单脱水 (Manifests Dump): 取出各类配置模板不部署 [调试向]。 | 获取集群元数据填入模板，生成静态的 CNI、DNS 的 Yaml 到 `/tmp/` 目录下供人工阅读审批。 |

## 3. Node Commands (个体计算单元指令集)

直接操作并解析 `client-go` 对 Kubernetes API 结果交互的集成测试。
包含指令: `node list`, `node get`, `node cordon`, `node uncordon`, `node drain`
相关参数: `--cluster (-c)`, `--kubeconfig`, `--output (-o)`, `--grace-period`, `--force`, `--ignore-daemonsets`, `--timeout`

| 用例 ID | 组合指令输入 | 测试意图与执行环境 | 终态断言与校验标准 |
|---|---|---|---|
| **NO-001** | `node list -c myc --kubeconfig /my/cf` | 拉取凭据读取节点宏观状态列表。 | 解析 API 响应，打印并格式化集群全部节点名称、IP 列表、Ready 状态及内核版本号。 |
| **NO-002** | `node get wk-1 -c myc -o json` | 拉取单一节点详单转特定格式。 | 工具解析并返吐出包括元数据、污点(Taints) 和各种节点健康状态(Conditions) 信息的 JSON 体。 |
| **NO-003** | `node cordon wk-1 -c myc` | 针对计算节点实施隔离拦截。 | 在对应节点的 K8s Spec 下挂出 `SchedulingDisabled` 的屏障标，阻止任何新业务流入。 |
| **NO-004** | `node uncordon wk-1 -c myc` | 解锁调度拦截。 | 取消对象属性中的不可调度标，该节点重归算力池接收分派。 |
| **NO-005** | `node drain wk-1 -c myc --force --ignore-daemonsets --grace-period=10` | 带覆盖标志位的**强力排障疏散令**。 | 忽略因 DaemonSet 管理带来的拒绝操作、强制终结那些缺少控制器的孤岛 Pod；给出 10 秒倒数后断开所有挂载将机器推入纯粹维护模式下线状态。 |

## 4. Certs Commands (安全认证层更新与核准)

对集群致命级 TLS 过期或更新的容灾管理。
包含指令: `certs renew`, `certs check-expiration`, `certs update`
包含子命令群: `all`, `kubernetes-ca`, `kubernetes-certs`, `etcd-ca`, `etcd-certs`

| 用例 ID | 组合指令输入 | 测试意图与执行环境 | 终态断言与校验标准 |
|---|---|---|---|
| **CE-001** | `certs renew all --yes` | 全量更新各系统安全通信凭证网络。 | K8s 与 ETCD 底座内的所有根信链与旁支节点证书悉数更换完毕；组件随后接令重加载，新证书被验证相互 TLS 连接有效。 |
| **CE-002** | `certs renew kubernetes-ca --yes` | 控制侧重塑：更新 K8s Root CA 与体系。 | 在 ETCD 服务完全不干预的情况下，专属于 Kubernetes 的核心 CA 重生，并且利用其批量更迭全部相干的业务端点信链。 |
| **CE-003** | `certs renew kubernetes-certs --yes` | 到期叶子证书签发延保。 | 保留原有的根证书结构（即原 CA 系统不乱），向 API/Kubelet 各末端颁发具有更长时效的新证书，验证该举动未造成当前集群系统崩断。 |
| **CE-004** | `certs check-expiration` | 全局日历审计。 | 工具自动化拉取机器上位于敏感存放域的 PEM，核算出并逐一打印证书创建至过期日期的期限余量或超时预警。 |

## 5. 存储仓、工具链及环境配套管理 (Registry, ISO, Tools, Config)

### 5.1 Registry & Images (私建内网资源库)
包含指令: `registry create`, `registry delete`, `images push`

| 用例 ID | 组合指令输入 | 测试意图与执行环境 | 终态断言与校验标准 |
|---|---|---|---|
| **RI-001** | `registry create --yes` | 纯孤岛隔离状态构筑内置 Registry。 | 挑中一台管控节点拉起具备超级认证 (Basic Auth) 身份体系的 Registry/Harbor 仓库池，同时开放指定监听服务。 |
| **RI-002** | `registry delete --yes` | 清洗与下架私库。 | 服务彻底停机、配置摘除及关联镜像目录卷被一并清零释放。 |
| **RI-003** | `images push` | [边缘测试] 极限并发流载入资源。 | 绕开不必要的安全策略（设定为 `tls-verify=false` 的受信主机），将海量的本地 Tar 打包镜像顺畅无阻推流存入特定内部前缀的库体中。 |

### 5.2 Download & ISO (全天候脱网建仓预备)
包含指令: `download`, `iso create`, `iso build`
相关参数: `--config (-f)`, `--output (-o)`

| 用例 ID | 组合指令输入 | 测试意图与执行环境 | 终态断言与校验标准 |
|---|---|---|---|
| **DI-001** | `download -f c.yaml -o /tmp/pkg` | 略去节点实测强行打包装箱 (用于跨平台制备弹药)。 | 工具跳过对主机集群实存性测试的 `Preflight` 模块，纯粹借助外部网络按需求（如指定的 amd64 及 arm64 双架构），大规模下载所有的 Docker 包、二进制源码及 RPM 到 `/tmp/pkg` 分级目录中妥善放置。 |
| **DI-002** | `iso create -f c.yaml -o release.iso` | 为离线化交付量身定制的无忧盘制备。 | 工具集调用基础层的 `xorriso` / `mkisofs` 处理机制，在提供给出的底层操作系统映像（Linux 基础）内硬灌入各种已准备好的物料及安装环境工具 (如 Grub 配置表)。所导出的 `.iso` 镜像具有启动后自动挂载集群装设流程的能力。 |
